## 担当範囲

### 認証（Authentication）

- Supabase Auth（JWT）の設定・管理
- ログイン/ログアウトフローの安全性確認
- トークンの管理（有効期限、リフレッシュ）

### 認可（Authorization / RBAC）

- ロール定義（HQ / Patrol / Exhibitor / Accounting / Property）
- `roles` / `user_roles` テーブルによるロール制御
- `user_organizations` テーブルによる団体境界の制御
- ロールに応じた画面・機能アクセスの制限

### RLS（Row Level Security）検証

- 各テーブルの RLS ポリシーが正しく機能しているか検証
- ロール別のアクセステスト（閲覧/作成/更新/削除）
- 権限外アクセスの防止確認

### 脆弱性対策

- OWASP Top 10 への対応
- XSS（クロスサイトスクリプティング）防止
- SQL インジェクション防止（Supabase のパラメータ化クエリを使用）
- コマンドインジェクション防止
- 外部入力のバリデーション

## RLS ポリシー検証チェックリスト

### 出展（Exhibitor）

- [ ] 自団体の連絡案件のみ閲覧できる
- [ ] 他団体の連絡案件は閲覧できない
- [ ] 自団体の連絡案件に追記できる
- [ ] 他団体の連絡案件には追記できない

### 本部（HQ）

- [ ] 全件閲覧できる
- [ ] 状態更新・担当割当ができる
- [ ] タスクの作成・割当ができる

### 会計（Accounting）

- [ ] notify_target=accounting の案件のみ閲覧できる
- [ ] 返信投稿・完了更新ができる
- [ ] 他の案件は閲覧できない

### 物品（Property）

- [ ] notify_target=property の案件のみ閲覧できる
- [ ] 返信投稿・完了更新ができる
- [ ] 他の案件は閲覧できない

### 巡回（Patrol）

- [ ] 未割当タスク＋自分担当のタスクが閲覧できる
- [ ] accept / complete ができる
- [ ] 他の巡回のタスクは変更できない

## 環境変数管理

### 必須

- API キー、パスワードは環境変数（`.env`）で管理する
- `.env` ファイルは `.gitignore` に含める
- `.env.example` にキー名のみ記載する（値は含めない）

### 禁止

- ソースコードへの API キーのハードコード
- ログへの機密情報の出力
- コミットへの `.env` ファイルの含有

---

## Storage セキュリティ

- バケット分離（`ticket_attachments`）
- 期限付き URL またはサイン付き URL を使用
- RLS でファイル参照の権限制御（連絡案件閲覧可能者のみ）

- 環境変数管理の確認結果

---

## 注意事項

- セキュリティの問題は最優先で対応する
- RLS は DB 担当と連携して設計・検証する
- 新しい機能追加時は、必ずセキュリティ観点でレビューする
- 外部ライブラリの脆弱性情報を定期的にチェックする
- 不要な権限を付与しない（最小権限の原則）
