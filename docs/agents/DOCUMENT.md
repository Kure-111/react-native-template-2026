## ドキュメント品質基準

### 必須項目

- 最終更新日を明記する
- 更新履歴を記録する
- 目次を付ける（長いドキュメント）
- コード例は実際に動くものを記載する

### 文書スタイル

- 日本語で記述する
- 技術用語は一貫した表記を使う
- 箇条書きを活用して読みやすくする
- テーブル形式を活用してデータを整理する

### 禁止事項

- 古い情報をそのまま残さない（更新 or 削除）
- 仕様書とコードの乖離を放置しない
- 曖昧な表現（「〜かもしれない」「おそらく」）を避ける

---

## 作業の進め方

1. 各担当の実装内容を確認する
2. 仕様書との差分を検出する
3. ドキュメントを更新する
4. 更新履歴を記録する
5. 他の担当にレビューを依頼する（必要に応じて）

---

## 出力物

- 仕様書の更新（`docs/`）
- API 仕様書
- 運用マニュアル
- トラブルシューティングガイドの更新
- データベースドキュメントの更新

---

## 注意事項

- ドキュメントの更新は実装と同期させる（遅延させない）
- 仕様変更があった場合は、関連する全ドキュメントを更新する
- コードコメント（JSDoc）とドキュメントの整合性を保つ
- 機密情報（APIキー、パスワード等）をドキュメントに含めない
