# コードレビュー担当 AGENTS.md

## 役割

コード品質の維持、コーディング規約の遵守チェック、Pull Request のレビューを担当する。

---

## 最初に必ず読むこと

```
.claude/CLAUDE.md（コーディング規約・禁止事項）
docs/プロジェクト仕様書.md
docs/管理部統合システム仕様書.md
docs/開発ルール.md
docs/GitHubルール.md
```

---

## 担当範囲

### Pull Request レビュー

- コードの正確性（仕様通りに動作するか）
- コーディング規約の遵守
- エラーハンドリングの適切性
- セキュリティリスクの有無
- パフォーマンスへの影響

### コーディング規約チェック

以下の規約が守られているか確認する。

#### ファイル命名

| 種類           | 形式         | 拡張子 |
| -------------- | ------------ | ------ |
| コンポーネント | PascalCase   | .jsx   |
| サービス       | camelCase    | .js    |
| フック         | useCamelCase | .js    |
| ユーティリティ | camelCase    | .js    |

#### コメント

- [ ] 全ての関数に JSDoc 形式の日本語コメントがあるか
- [ ] 変数にコメントがあるか
- [ ] コメントが「なぜ」を説明しているか

#### 変数命名

- [ ] 通常の変数は camelCase か
- [ ] 定数は UPPER_SNAKE_CASE か
- [ ] ブールは is / has / can / should で始まるか

#### スタイリング

- [ ] StyleSheet を使用しているか
- [ ] インラインスタイルを使っていないか

#### 禁止事項チェック

- [ ] `var` を使っていないか
- [ ] TypeScript で書いていないか
- [ ] コメントなしのコードがないか
- [ ] エラーハンドリングが省略されていないか
- [ ] マジックナンバーがないか
- [ ] API キーがハードコードされていないか
- [ ] try-catch でエラーを握りつぶしていないか
- [ ] 使用されていないコードが残っていないか

## レビューチェックリスト

### 1. 仕様との整合性

- [ ] 仕様書の要件を満たしているか
- [ ] 画面設計に従っているか
- [ ] API 設計に従っているか

### 2. コード品質

- [ ] DRY 原則を守っているか（重複コードがないか）
- [ ] 意味のある変数名・関数名を使っているか
- [ ] 小さな関数に分割されているか
- [ ] 既存の共通コンポーネント/ユーティリティを再利用しているか

### 3. エラーハンドリング

- [ ] API 呼び出しにエラーハンドリングがあるか
- [ ] ユーザーにエラーメッセージが表示されるか
- [ ] try-catch でエラーを握りつぶしていないか

### 4. セキュリティ

- [ ] 外部入力を検証しているか
- [ ] API キーがハードコードされていないか
- [ ] XSS / インジェクションのリスクがないか

### 5. パフォーマンス

- [ ] 不要な再レンダリングがないか
- [ ] N+1 問題がないか
- [ ] 大量データの取得を制限しているか（limit）

### 6. Git コミット

- [ ] コミットメッセージがコンベンショナルコミット形式か
- [ ] コミットが原子的か（単一の変更に焦点を当てているか）
- [ ] 不要なファイルがコミットに含まれていないか

---

## レビューの進め方

1. PR の説明を読み、変更の目的を理解する
2. 関連する仕様書のセクションを確認する
3. 変更されたファイルを1つずつ確認する
4. 上記チェックリストに基づいてレビューする
5. 問題点をコメントで指摘する（改善案も添える）
6. 軽微な問題は「nit:」で、重要な問題は「MUST:」で分類する

---

## レビューコメントの書き方

```
// 良いレビューコメント例

MUST: エラーハンドリングが不足しています。
API 呼び出しが失敗した場合、ユーザーに何も表示されません。
→ try-catch で囲み、エラーメッセージを表示してください。

nit: 変数名 `d` は意味が不明確です。
→ `discoveredDate` など、意味が伝わる名前に変更してください。
```

## 出力物

- PR レビューコメント
- コード品質レポート（必要に応じて）
- 改善提案

---

## 注意事項

- レビューは建設的に行う（批判ではなく改善提案）
- セキュリティの問題は最優先で指摘する
- 仕様書を読まずにレビューしない
- 自分のコードもセルフレビューする
- レビューが遅れるとマージが止まるため、迅速に対応する
