# 整理券システム テーブル解説書

このドキュメントは、別プロジェクト（配布状況確認システム）を開発するAI向けに、Supabaseのテーブル構造とUUIDの依存関係を解説するものです。

---

## 1. システム概要

大学祭の整理券発券システム。2種類の企画タイプに対応しています。

| 企画タイプ | 説明 | 使用テーブル |
|-----------|------|-------------|
| **時間枠定員制** (`time_slot`) | 時間枠ごとに定員がある（例：10:00〜11:00で50名） | events, event_dates, time_slots, tickets |
| **順次案内制** (`sequential`) | 時間枠なし、空きが出次第案内 | events, event_dates, tickets, call_status |

## 3. 各テーブルの詳細

### 3.1 events（企画テーブル）

**役割**: 企画の基本情報を管理する親テーブル

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| **id** | UUID | 主キー | 自動生成 |
| name | TEXT | 企画名 | 例: "お化け屋敷" |
| location | TEXT | 企画場所 | 例: "1号館2階" |
| **type** | TEXT | 企画タイプ | `'time_slot'` または `'sequential'` |
| capacity_per_slot | INTEGER | 1枠あたりの定員 | **time_slotの場合のみ必須** |
| slot_duration_minutes | INTEGER | 1枠の時間（分） | **time_slotの場合のみ必須** |
| estimated_wait_minutes | INTEGER | 1番号あたりの待ち時間（分） | **sequentialの場合のみ必須** |
| created_at | TIMESTAMP | 作成日時 | |
| updated_at | TIMESTAMP | 更新日時 | |

**重要**: `type`カラムの値によって、使用するカラムと関連テーブルが変わる

```sql
-- 時間枠定員制の例
INSERT INTO events (name, location, type, capacity_per_slot, slot_duration_minutes)
VALUES ('お化け屋敷', '1号館2階', 'time_slot', 50, 60);

-- 順次案内制の例
INSERT INTO events (name, location, type, estimated_wait_minutes)
VALUES ('模擬店A', '中庭', 'sequential', 5);
```

---

### 3.2 event_dates（企画開催日テーブル）

**役割**: 企画がどの日に開催されるかを管理

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| **id** | UUID | 主キー | 自動生成 |
| **event_id** | UUID | 企画ID | **→ events.id への外部キー** |
| date | DATE | 開催日 | 例: '2024-11-02' |
| status | TEXT | ステータス | 下記参照 |
| next_ticket_number | INTEGER | 次の整理番号 | **順次案内制で使用**（デフォルト: 1） |
| created_at | TIMESTAMP | 作成日時 | |
| updated_at | TIMESTAMP | 更新日時 | |

**ステータス値**:
| 値 | 意味 |
|----|------|
| `not_started` | 未発券（まだ開始していない） |
| `active` | 発券中 |
| `paused` | 発券停止中（一時停止） |
| `full` | 満員 |
| `ended` | 発券終了 |

**UUID依存関係**:
```
events.id ──< event_dates.event_id
  (1)              (多)
```

1つの企画(events)は複数の開催日(event_dates)を持つ

---

### 3.3 time_slots（時間枠テーブル）

**役割**: 時間枠定員制の企画で、各時間枠を管理

**注意**: `events.type = 'time_slot'` の場合のみ使用

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| **id** | UUID | 主キー | 自動生成 |
| **event_id** | UUID | 企画ID | **→ events.id への外部キー** |
| **event_date_id** | UUID | 企画開催日ID | **→ event_dates.id への外部キー** |
| start_time | TIME | 開始時刻 | 例: '10:00:00' |
| end_time | TIME | 終了時刻 | 例: '11:00:00' |
| status | TEXT | ステータス | event_datesと同じ値 |
| current_count | INTEGER | 現在の発券数 | デフォルト: 0 |
| **start_ticket_number** | INTEGER | 開始整理番号 | 例: 1, 51, 101... |
| created_at | TIMESTAMP | 作成日時 | |
| updated_at | TIMESTAMP | 更新日時 | |

**UUID依存関係**:
```
events.id ──────────< time_slots.event_id
event_dates.id ─────< time_slots.event_date_id
```

**整理番号の計算**:
```
整理番号 = start_ticket_number + current_count

例: 定員50人の場合
- 10:00〜11:00枠: start_ticket_number = 1   → 1〜50
- 11:00〜12:00枠: start_ticket_number = 51  → 51〜100
- 12:00〜13:00枠: start_ticket_number = 101 → 101〜150
```

---

### 3.4 tickets（整理券テーブル）

**役割**: 発行された全ての整理券を記録

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| **id** | UUID | 主キー | 自動生成 |
| **event_id** | UUID | 企画ID | **→ events.id への外部キー** |
| **event_date_id** | UUID | 企画開催日ID | **→ event_dates.id への外部キー** |
| **time_slot_id** | UUID | 時間枠ID | **→ time_slots.id への外部キー、NULL許可** |
| ticket_number | INTEGER | 整理番号 | 例: 1, 2, 51, 52... |
| medium_type | TEXT | 媒体種類 | `'paper'` または `'digital'` |
| qr_token | TEXT | QRトークン | **digitalの場合のみ**、UNIQUE |
| issued_at | TIMESTAMP | 発券日時 | |
| created_at | TIMESTAMP | 作成日時 | |

**UUID依存関係（最重要）**:
```
events.id ──────────< tickets.event_id       (必須)
event_dates.id ─────< tickets.event_date_id  (必須)
time_slots.id ──────< tickets.time_slot_id   (NULL許可)
```

**time_slot_idの判定**:
- `time_slot_id IS NOT NULL` → 時間枠定員制の整理券
- `time_slot_id IS NULL` → 順次案内制の整理券

### 3.5 call_status（呼び出し状態テーブル）

**役割**: 順次案内制の企画で、現在何番まで呼び出しているかを管理

**注意**: `events.type = 'sequential'` の場合のみ使用

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| **id** | UUID | 主キー | 自動生成 |
| **event_id** | UUID | 企画ID | **→ events.id への外部キー** |
| **event_date_id** | UUID | 企画開催日ID | **→ event_dates.id への外部キー** |
| current_call_number | INTEGER | 現在の呼び出し番号 | デフォルト: 0 |
| updated_at | TIMESTAMP | 更新日時 | |

**UNIQUE制約**: `(event_id, event_date_id)` の組み合わせは1つのみ

**UUID依存関係**:
```
events.id ──────────< call_status.event_id
event_dates.id ─────< call_status.event_date_id
```

### 3.6 ticket_logs（発券ログテーブル）

**役割**: 発券履歴のログを記録

| カラム | 型 | 説明 | 備考 |
|--------|-----|------|------|
| **id** | UUID | 主キー | 自動生成 |
| **ticket_id** | UUID | 整理券ID | **→ tickets.id への外部キー** |
| action | TEXT | アクション | デフォルト: 'issued' |
| performed_at | TIMESTAMP | 実行日時 | |
| details | JSONB | 詳細情報 | JSON形式で追加情報を保存 |

**UUID依存関係**:
```

```

## 5. Supabase JavaScript Client でのデータ取得例

このシステムでは生SQLではなく、**Supabase JavaScript Client** のクエリビルダーを使用しています。

### 5.1 全企画と開催日を取得

```javascript
const { data, error } = await supabase
  .from('events')
  .select(`
    *,
    event_dates (
      id,
      date,
      status,
      next_ticket_number
    )
  `)
  .order('created_at', { ascending: false });

// 取得後、日付を昇順でソート
if (data) {
  data.forEach(event => {
    if (event.event_dates) {
      event.event_dates.sort((a, b) => new Date(a.date) - new Date(b.date));
    }
  });
}
```

### 5.2 企画をIDで取得（時間枠も含む）

```javascript
const { data, error } = await supabase
```

## 6. Supabase Clientのクエリ構文まとめ

| 操作 | メソッド | 例 |
|------|----------|-----|
| 全件取得 | `.select('*')` | `supabase.from('events').select('*')` |
| 特定カラム | `.select('id, name')` | `supabase.from('events').select('id, name')` |
| リレーション | `.select('*, テーブル名 (カラム)')` | 上記5.5参照 |
| 条件絞り込み | `.eq('カラム', 値)` | `.eq('id', 'uuid...')` |
| 複数条件 | `.in('カラム', [値1, 値2])` | `.in('status', ['active', 'paused'])` |
| ソート | `.order('カラム', { ascending: true/false })` | `.order('date', { ascending: true })` |
| 1件取得 | `.single()` | 必ず1件返る場合に使用 |
| INSERT | `.insert({ ... }).select()` | `.insert({ name: 'test' }).select()` |
| UPDATE | `.update({ ... }).eq(...).select()` | `.update({ status: 'active' }).eq('id', id)` |
| DELETE | `.delete().eq(...)` | `.delete().eq('id', id)` |

## 7. 参考：生SQLクエリ例

生SQLで書くとこのような状態です。テーブル構造の理解用に記載しています。

### 7.1 全企画と開催日を取得

```sql
SELECT
    e.id AS event_id,
    e.name AS event_name,
    e.type AS event_type,
    e.capacity_per_slot,
    ed.id AS event_date_id,
    ed.date,
    ed.status,
    ed.next_ticket_number
FROM events e
LEFT JOIN event_dates ed ON e.id = ed.event_id
ORDER BY e.created_at DESC, ed.date ASC;
```

### 7.2 時間枠定員制の発券状況を取得

```sql
SELECT
    e.name AS event_name,
    ed.date,
    ts.start_time,
    ts.end_time,
    ts.current_count,
    e.capacity_per_slot,
    ts.status,
    -- 発券率
    ROUND(ts.current_count * 100.0 / e.capacity_per_slot, 1) AS fill_rate
FROM events e
JOIN event_dates ed ON e.id = ed.event_id
JOIN time_slots ts ON ed.id = ts.event_date_id
WHERE e.type = 'time_slot'
ORDER BY ed.date, ts.start_time;
```

### 7.3 順次案内制の発券・呼び出し状況を取得

```sql
SELECT
    e.name AS event_name,
    ed.date,
    ed.next_ticket_number - 1 AS current_ticket_number,  -- 発券済み枚数
    cs.current_call_number,                               -- 呼び出し済み番号
    (ed.next_ticket_number - 1 - cs.current_call_number) AS waiting_count,  -- 待ち人数
    e.estimated_wait_minutes,
    (ed.next_ticket_number - 1 - cs.current_call_number) * e.estimated_wait_minutes AS estimated_wait  -- 推定待ち時間
FROM events e
JOIN event_dates ed ON e.id = ed.event_id
LEFT JOIN call_status cs ON e.id = cs.event_id AND ed.id = cs.event_date_id
WHERE e.type = 'sequential';
```

### 7.4 特定の整理券を取得（QRトークンから）

```sql
SELECT
    t.ticket_number,
    t.medium_type,
    e.name AS event_name,
    e.location,
    ed.date,
    ts.start_time,
    ts.end_time
FROM tickets t
JOIN events e ON t.event_id = e.id
JOIN event_dates ed ON t.event_date_id = ed.id
LEFT JOIN time_slots ts ON t.time_slot_id = ts.id
WHERE t.qr_token = '指定のトークン';
```

### 7.5 日付別・企画別の発券数集計

```sql
SELECT
    e.name AS event_name,
    e.type AS event_type,
    ed.date,
    COUNT(t.id) AS total_tickets,
    SUM(CASE WHEN t.medium_type = 'paper' THEN 1 ELSE 0 END) AS paper_count,
    SUM(CASE WHEN t.medium_type = 'digital' THEN 1 ELSE 0 END) AS digital_count
FROM events e
JOIN event_dates ed ON e.id = ed.event_id
LEFT JOIN tickets t ON ed.id = t.event_date_id
GROUP BY e.id, e.name, e.type, ed.id, ed.date
ORDER BY ed.date, e.name;
```

## 8. 企画タイプ別のデータフロー

### 8.1 時間枠定員制 (`type = 'time_slot'`)

```
企画登録時:
  events (INSERT) → event_dates (INSERT) → time_slots (INSERT, 複数)

発券時:
  1. events.id, event_dates.id, time_slots.id を指定
  2. time_slots.current_count を確認（定員チェック）
  3. tickets (INSERT)
  4. time_slots.current_count を +1
  5. 定員到達時: time_slots.status = 'full' に更新

データ取得時:
  events → event_dates → time_slots → tickets の順にJOIN
```

### 8.2 順次案内制 (`type = 'sequential'`)

```
企画登録時:
  events (INSERT) → event_dates (INSERT) → call_status (INSERT)

発券時:
  1. events.id, event_dates.id を指定（time_slot_id は NULL）
  2. event_dates.next_ticket_number を取得
  3. tickets (INSERT)
  4. event_dates.next_ticket_number を +1

呼び出し時:
  call_status.current_call_number を更新

データ取得時:
  events → event_dates → tickets (time_slot_id IS NULL)
  events → event_dates → call_status
```

## 10. 注意事項

### RLS（Row Level Security）

- 全テーブルでRLSが有効化されている
- 現在は全アクセス許可のポリシーが設定されている
- 本番環境では適切なポリシーに変更すること

### Realtime

- `call_status` テーブルはRealtimeが有効
- 呼び出し番号の変更をリアルタイムで受信可能

### CASCADE削除

- 全ての外部キーに `ON DELETE CASCADE` が設定されている
- 親レコードを削除すると、関連する子レコードも自動削除される

### Q: 特定日の発券状況を取得するには？

```javascript
// 時間枠定員制
const { data } = await supabase
  .from('time_slots')
  .select(`
    *,
    event_dates!inner (date),
    events!inner (capacity_per_slot)
  `)
  .eq('event_dates.date', '2024-11-02');

// 順次案内制
const { data } = await supabase
  .from('event_dates')
  .select(`
    next_ticket_number,
    events!inner (type),
    call_status (current_call_number)
  `)
  .eq('date', '2024-11-02')
  .eq('events.type', 'sequential');

// 発券済み枚数: next_ticket_number - 1
// 待ち人数: (next_ticket_number - 1) - current_call_number
```
