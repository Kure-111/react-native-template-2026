# プロジェクト仕様書

## プロジェクト概要

生駒祭 2026 で使用する ERP（Enterprise Resource Planning）システム。
Web アプリケーションとして動作し、PC とスマートフォンの両方に対応する。

### 技術スタック

- **フロントエンド:** React Native (Expo) Web
- **バックエンド:** Supabase
- **その他:** Google Apps Script（必要に応じて）
- **使用言語:** JavaScript（TypeScript ではない）

### 対応プラットフォーム

- **PC:** 768px 以上の画面幅
- **スマートフォン:** 768px 未満の画面幅

## 機能要件

### 項目1〜10

各機能の詳細は実装時に定義する。
左にサイドバーがあり、項目を切り替えられる。のちにsupabaseを用いてユーザごとのアクセス制御を設ける。スマホ用はサイドバーではなくハンバーガーメニューとして集約されている。いずれの項目もフェイクページである。このフェイクページはエラーなどでアクセスできなかった際に表示されるものである。

### 項目11：当日部員シフト確認

ログインユーザーの当日部員シフトをパーソナライズして表示する機能。

#### 目的

従来の表形式で全体シフトを閲覧していた方式から、ERPのログイン機能を活かしたパーソナライズ表示に変更。各ユーザーが自分のシフトのみを確認できる。

#### 主な機能

1. **日付選択**
   - カレンダーまたは日付ピッカーで日付を自由に選択可能
   - 過去・未来の日付も制限なく選択可能

2. **シフト表示**
   - 選択した日付の自分のシフトを表示
   - 時間帯、担当エリア（場所）を表示
   - 複数の団体に所属している場合、全ての団体のシフトを合算して表示
   - シフトがない場合は「この日はシフトがありません」と表示
   - 背景色情報は取得できないため、団体ごとに固定色を割り当て
     - テキストは黒で視認性を確保

3. **データ取得方法**
   - Google Spreadsheetsから直接CSV形式で取得（GASデプロイ不要、シートidは環境変数として.envに保存『EXPO_PUBLIC_SHIFT_SPREADSHEET_ID』）
   - スプレッドシートは「m月d日」という名前のシートで日付ごとに管理（ゼロ埋めなし）
   - スプレッドシートを「リンクを知っている全員が閲覧可能」に設定
   - ユーザーの所属団体（rolesのname）に基づいてシートから該当行を検索
   - ユーザー名（user_profiles.name）でシフトを特定

#### データフロー

```
1. ユーザーが日付を選択（例: 11月3日）
2. フロントエンドがGoogle SheetsにCSV取得リクエスト（シート名: "11月3日"）
3. Google Sheetsがスプレッドシートから以下を取得:
   - セルの値（CSV形式）
4. フロントエンドがCSVをパースして2次元配列に変換
5. ユーザーの所属団体とユーザー名で該当行をフィルタリング
6. 該当するシフトを抽出:
   - セルに文字がある = シフトあり（エリア名を取得）
   - 団体ごとに固定色を割り当て
7. シフト情報を時系列順に並べて表示:
   - 時間帯、エリア名
   - 固定色を薄くしてカード背景に適用
```

#### スプレッドシート構造

**シート構成:**
- シート名：「m月d日」形式（例: 「11月3日」「12月25日」）※ゼロ埋めなし
- 団体ごとにブロックが分かれている
- 各ブロックは団体名行 + ヘッダー行 + データ行で構成

**団体名行:**
```
事務部 (総数:7, 確保:0, 割当可:7)
```
- 団体名の後に括弧で人数情報が記載されている
- この括弧部分は無視する

**ヘッダー行:**
```
| 名前 | 07:30-08:00 | 08:00-08:30 | 08:30-09:00 | ... |
```
- 1列目: 「名前」
- 2列目以降: 時間帯（30分刻み）

**データ行:**
- 1列目: ユーザー名（Supabase の user_profiles.name と照合）
- 2列目以降: シフト情報
  - **空白セル**: シフトなし
  - **文字入りセル**: シフトあり
    - セル内テキスト = エリア名（例: 「警備」「テント」など）


#### 複数団体所属時の挙動

ユーザーが複数の団体に所属している場合（例: 事務部 + 会計部）:
- 両方の団体のシフトを取得
- 時系列順に並べて表示
- 例:
  - 11:00〜12:00 エリアA
  - 13:00〜14:00 エリアB

#### 制限事項

- データの読み取りのみ（編集機能なし）
- 学内認証の制約により、書き込み操作は行わない

#### アクセス権限

現時点では特に制限なし。将来的に祭実長権限を持つユーザのみ申請できるボタンがあるなど、ロールベースで制御可能。

## エラーハンドリング設計

### Error Boundary アーキテクチャ

各画面は Error Boundary でラップされており、個別の画面でエラーが発生しても他の画面に影響を与えない。

#### 構成

```
DrawerNavigator
  └── ScreenErrorBoundary（画面ごと）
        └── ItemXScreen
              └── PlaceholderContent
```

#### コンポーネント

| コンポーネント | 役割 |
|--------------|------|
| ScreenErrorBoundary | 画面単位のエラーを捕捉し、フォールバックUIを表示 |
| PlaceholderContent | 開発中画面およびエラー時のフォールバック表示 |

#### 動作フロー

1. **正常時:** 各 ItemXScreen が PlaceholderContent を表示（開発中）
2. **エラー発生時:** ScreenErrorBoundary がエラーを捕捉
3. **フォールバック:** PlaceholderContent をエラーモードで表示
4. **リトライ:** 「ホームに戻る」ボタンで Item1 に遷移可能

#### エラーログ

エラー発生時、以下の情報がコンソールに出力される。

- 画面名（screenName）
- エラーメッセージ
- コンポーネントスタック
- タイムスタンプ

## 認証システム設計

### 認証方式

**メールアドレス + パスワード認証**
- Supabase Auth を使用
- JWT ベースのセッション管理
- Row Level Security (RLS) による権限管理

### ログインフロー

1. ユーザーがメールアドレスとパスワードを入力
2. Supabase Auth API でログイン認証
3. 認証成功後、user_profiles と user_roles を取得
4. AuthContext に認証状態を保存
5. メイン画面に遷移

### ユーザー登録方法

**管理者による招待方式（推奨）**
- 管理者が Google スプレッドシートに名簿を入力
- GAS（Google Apps Script）が Supabase Auth API を呼び出し
- 自動的にユーザーアカウントを作成
- 仮パスワードを生成し、ユーザーに通知
- 初回ログイン後、ユーザーがパスワードを変更

## データベース設計

### テーブル構造

Supabase を使用。以下の3テーブルで構成。

#### 1. user_profiles テーブル

ユーザーのプロフィール情報を管理。

| カラム名 | 型 | 説明 | 制約 |
|---------|-----|------|------|
| id | UUID | 主キー | PRIMARY KEY |
| user_id | UUID | auth.users への外部キー | UNIQUE, NOT NULL, REFERENCES auth.users(id) ON DELETE CASCADE |
| name | TEXT | ユーザー名 | NOT NULL |
| organization | TEXT | 所属団体 | |
| discord_id | TEXT | Discord ユーザーID | UNIQUE |
| created_at | TIMESTAMPTZ | 作成日時 | DEFAULT NOW() |
| updated_at | TIMESTAMPTZ | 更新日時 | DEFAULT NOW() |

**インデックス:**
```sql
CREATE INDEX idx_user_profiles_user_id ON user_profiles(user_id);
CREATE INDEX idx_user_profiles_discord_id ON user_profiles(discord_id);
```

#### 2. roles テーブル

役職（ロール）の定義を管理。

| カラム名 | 型 | 説明 | 制約 |
|---------|-----|------|------|
| id | UUID | 主キー | PRIMARY KEY |
| name | TEXT | ロール名（システム用） | UNIQUE, NOT NULL |
| display_name | TEXT | 表示名 | NOT NULL |
| description | TEXT | 説明 | |
| permissions | JSONB | 権限設定 | DEFAULT '{}' |
| discord_role_id | TEXT | Discord ロールID | |
| discord_guild_id | TEXT | Discord サーバーID | |
| created_at | TIMESTAMPTZ | 作成日時 | DEFAULT NOW() |
| updated_at | TIMESTAMPTZ | 更新日時 | DEFAULT NOW() |

**permissions の JSONB 構造:**
```json
{
  "screens": ["item1", "item2", "item3"],
  "features": {
    "item1": ["view", "edit", "delete"],
    "item2": ["view"]
  }
}
```

#### 3. user_roles テーブル

ユーザーとロールの紐付けを管理（多対多の中間テーブル）。

| カラム名 | 型 | 説明 | 制約 |
|---------|-----|------|------|
| id | UUID | 主キー | PRIMARY KEY |
| user_id | UUID | auth.users への外部キー | NOT NULL, REFERENCES auth.users(id) ON DELETE CASCADE |
| role_id | UUID | roles への外部キー | NOT NULL, REFERENCES roles(id) ON DELETE CASCADE |
| created_at | TIMESTAMPTZ | 作成日時 | DEFAULT NOW() |

**制約:**
```sql
UNIQUE(user_id, role_id)
```

**インデックス:**
```sql
CREATE INDEX idx_user_roles_user_id ON user_roles(user_id);
CREATE INDEX idx_user_roles_role_id ON user_roles(role_id);
```

### Row Level Security (RLS) ポリシー

**user_profiles テーブル:**
```sql
-- 自分のプロフィールは読み取り可能
CREATE POLICY "Users can read own profile" ON user_profiles
  FOR SELECT USING (auth.uid() = user_id);

-- 自分のプロフィールは更新可能
CREATE POLICY "Users can update own profile" ON user_profiles
  FOR UPDATE USING (auth.uid() = user_id);
```

**user_roles テーブル:**
```sql
-- 自分のロールは読み取り可能
CREATE POLICY "Users can read own roles" ON user_roles
  FOR SELECT USING (auth.uid() = user_id);
```

**roles テーブル:**
```sql
-- 全員がロール定義を読み取り可能
CREATE POLICY "Anyone can read roles" ON roles
  FOR SELECT TO authenticated USING (true);
```

## API 設計

### Supabase Auth API

**ログイン:**
```javascript
supabase.auth.signInWithPassword({ email, password })
```

**ログアウト:**
```javascript
supabase.auth.signOut()
```

**セッション取得:**
```javascript
supabase.auth.getSession()
```

**認証状態監視:**
```javascript
supabase.auth.onAuthStateChange((event, session) => {})
```

### Supabase Database API

**ユーザープロフィール取得:**
```javascript
supabase
  .from('user_profiles')
  .select('*')
  .eq('user_id', userId)
  .single()
```

**ユーザーロール取得:**
```javascript
supabase
  .from('user_roles')
  .select('*, roles(*)')
  .eq('user_id', userId)
```

### Google Spreadsheet CSV取得API

**エンドポイント:**
```
https://docs.google.com/spreadsheets/d/{SPREADSHEET_ID}/gviz/tq?tqx=out:csv&sheet={sheetName}
```

**環境変数:** `EXPO_PUBLIC_SHIFT_SPREADSHEET_ID` でシフト用スプレッドシートIDを指定

**前提条件:**
- スプレッドシートを「リンクを知っている全員が閲覧可能」に設定

**レスポンス形式:**
CSV形式のテキストデータ
```csv
名前,07:30-08:00,08:00-08:30,...
部員A,オレンジ,,...
部員B,,黄色,...
```

**注意:**
- CSV形式のため、セルの背景色は取得できません
- シフトカードの色は団体ごとに固定色を割り当てます
- ダブルクォートで囲まれたセル値にはカンマが含まれる場合があります

   - Web App Manifestの設定
   - ホーム画面への追加機能
   - スタンドアロンモードでの起動（ブラウザのUIを非表示）
   - アプリアイコン、スプラッシュスクリーンの表示

2. **プッシュ通知**
   - 重要な更新や通知をユーザーに届ける
   - バックグラウンドでの通知受信
   - 通知のクリックでアプリの該当画面へ遷移

3. **レスポンシブ対応**
   - PC（768px以上）とスマートフォン（768px未満）の両方に最適化
   - タッチ操作に対応したUI/UX

4. **パフォーマンス最適化**
   - 高速な初期読み込み
   - プリキャッシュによる即座の画面表示
   - 効率的なリソース管理

#### 実装の技術要素

- **Expo EAS Build**: PWAビルドの生成
- **Workbox**: Service Workerの管理とキャッシュ戦略
- **Web App Manifest**: アプリのメタデータとアイコン設定
- **Push API**: プッシュ通知機能
- **Cache API**: オフライン対応のためのリソースキャッシュ
- **IndexedDB**: オフラインデータの永続化

#### PWA要件

- HTTPSでの配信（必須）
- Service Workerの登録
- manifest.jsonの配置
- アイコン（複数サイズ）の用意
- Lighthouseスコアで90点以上を目指す
