## プロジェクト概要

### 目的

既存の当日部員シフト確認画面（`jimu-shift`）に以下の 2 つの機能を追加する。

1. **シフト変更申請機能** — 祭実長・部長権限を持つユーザーが、自団体の部員シフトを確認し、部員間のシフト変更を「事務部」ロールのユーザーに申請・通知できる。事務部は外部サービスで実際のシフト表を修正した後、完了フィードバック通知を申請者に返す。
2. **シフトリマインド通知機能** — シフト開始 20 分前に、該当ユーザーへリマインド通知を自動送信する。

### 背景

- 当日のシフト変更は現在口頭やチャットで行われており、事務部への伝達漏れや対応の抜けが発生している。
- シフト時間を忘れて遅刻するケースが発生しており、自動リマインドの仕組みが必要。

### スケジュール

- 開発開始：2026/02/23
- リリース予定：未定

## 対象ユーザー

### 祭実長・部長

- **いつ使うか：** 当日、所属団体の部員がシフト変更を希望したとき
- **どういうときに使うか：** 「この時間行けなくなったから代わって」という部員間のシフト入れ替えを事務部に申請するとき

### 事務部

- **いつ使うか：** シフト変更申請通知を受け取ったとき
- **どういうときに使うか：** 外部シフト管理サービスで変更を反映した後、申請者に完了フィードバックを返すとき

### 全ユーザー（当日部員）

- **いつ使うか：** 自分のシフト開始 20 分前
- **どういうときに使うか：** リマインド通知を受け取り、シフト時間と配置場所を確認するとき

## 機能要件

### 機能 1：シフト変更申請

#### 1-1. 団体シフト一覧表示（祭実長・部長のみ）

**概要：**
祭実長・部長権限を持つユーザーが、自分の所属団体の全部員シフトをグリッド形式で閲覧できる。

**詳細仕様：**

- 既存の「当日部員」画面にタブを追加し、「マイシフト」（既存）と「シフト変更申請」を切り替え可能にする。
- 「シフト変更申請」タブは `祭実長` または `部長` ロールを持つユーザーにのみ表示する。
- 所属団体の判定は `roles` テーブルを使用する。祭実長・部長ロールと同時に保持している団体ロール（例：`情祭`、`営祭` 等）で団体を特定する。
- 同一日のシフトデータを GAS API から取得し、団体ブロック内の全メンバーのシフトをグリッド形式で表示する。
- グリッドの行はメンバー名、列は時間帯（30 分刻み）、セルにはエリア名が表示される。
- 日付選択は既存と同様、祭り期間内のみ切り替え可能。

---

#### 1-2. シフト変更の選択操作

**概要：**
祭実長・部長がグリッド上でシフトセルを選択し、同じ時間帯の他メンバーに交代を依頼する変更内容を確定する。

**詳細仕様：**

- 操作は 2 ステップで行う：
  1. **「交代してほしいシフトの選択」** — グリッド上のシフトが入っているセルをタップして選択（選択中はハイライト表示）。
  2. **「交代相手の選択」** — **同じ時間帯の**別メンバーのセルをタップして交代先を指定。
- 変更は**同日・同一時間帯内のみ**に限定する。異なる時間帯への変更は不可。
  - 交代元を選択後、異なる時間帯の列はグレーアウト表示され選択できない。
- 交代先が空きセルの場合：交代元のメンバーが空き枠を担当する形で申請（移動パターン）。
- 交代先に既にシフトがある場合：2 人のシフトを入れ替える形で申請（交換パターン）。
- 選択後、確認モーダルを表示し、変更内容のプレビュー（誰と誰が同じ時間帯のシフトを交代するか）を表示する。
- 確認モーダルで「申請」ボタンを押すと事務部への通知が送信される。
- 申請者は任意で備考（最大 200 文字）を入力できる。

**ブロック選択仕様（連続同一エリアのまとめ選択）：**

- 同一メンバーが連続する時間帯に**同じエリア名**でシフトが入っている場合、それらを「1 ブロック」として扱う。
  - 例：Aさん が `8:00-8:30 ステッカー` と `8:30-9:00 ステッカー` を持つ場合、どちらか一方をタップすると両方が自動選択される。
- 交代元がブロック選択された場合、交代先もブロックと同じコマ数・時間帯を自動補完する。
  - 交代先メンバーの対象時間帯のセルが 1 件ずつ選択されていても、自動的に同じ時間帯範囲全体を宛先として補完する。
- `source_time_slot` / `destination_time_slot` には合成した時間帯文字列を格納する（例：`"8:00-9:00"`）。

---

#### 1-3. シフト変更申請通知の送信

**概要：**
変更内容を事務部ロールのユーザー全員に通知する。

**詳細仕様：**

- 既存の通知サービス（`notificationService.js`）の `sendNotificationToRole` を使用する。
- 送信先は `事務部` ロールのユーザー全員。
- 通知タイトル：`「シフト変更申請」`
- 通知本文：変更内容の要約（例：`「情祭 - 田中さん(10:00-10:30 ステッカー) ↔ 山田さん(11:00-11:30 渉外部B館前)」`）
- `metadata` に以下を格納する：

```json
{
  "type": "shift_change_request",
  "request_id": "uuid",
  "date": "2026-11-03",
  "organization_name": "情祭",
  "source": {
    "member_name": "田中",
    "time_slot": "10:00-10:30",
    "area_name": "ステッカー"
  },
  "destination": {
    "member_name": "山田",
    "time_slot": "11:00-11:30",
    "area_name": "渉外部B館前"
  },
  "requester_user_id": "uuid"
}
```

- 申請データは `shift_change_requests` テーブルに保存する。
- 送信成功後、申請者には「申請を送信しました」のフィードバックを表示する。
- 送信失敗時はエラーメッセージを表示する。

---

#### 1-4. 事務部向け申請管理・フィードバック通知

**概要：**
事務部は専用の「変更申請管理」タブで全申請を一覧確認し、承認・却下のフィードバックを申請者に返す。

**詳細仕様：**

- 事務部ロールを持つユーザーに対し、「当日部員」画面に「変更申請管理」タブを追加表示する。
- 申請一覧はフィルタータブ（未対応 / 対応済み / 全て）で絞り込み可能。デフォルトは「未対応」表示。
- 申請カードには以下を表示する：
  - ステータスバッジ（未対応 / 完了 / 却下）
  - 団体名・シフト日付
  - 変更種別（交換 / 移動）と変更内容
  - 申請者備考（入力がある場合）
  - 申請日時
- 申請カードをタップすると「対応モーダル」を表示する。

**対応モーダル（ShiftChangeResponseModal）：**

- 申請詳細（変更内容・申請者備考）を表示する。
- 事務部担当者は対応備考（必須、最大 500 文字）を入力する。
- 「承認」または「却下」ボタンを押すと、インライン確認エリアを表示する（`Alert.alert` は使用しない）。
  - 確認エリアに「戻る」と「完了にする / 却下する」ボタンを配置する。
- 確定後、申請者に通知を送信し、ステータスを更新する。
- 既に対応済みの申請は操作ボタンを非表示にし、対応内容を閲覧のみ可能とする。

**完了通知（承認時）：**

- 送信先：申請者（`requester_user_id`）
- 完了通知タイトル：`「シフト変更完了」`
- 完了通知本文：`「申請されたシフト変更が反映されました。」` + 変更内容の要約 + 対応備考
- 完了通知の `metadata`：

```json
{
  "type": "shift_change_completed",
  "request_id": "元の申請ID",
  "completed_by": "事務部ユーザーID"
}
```

**却下通知（却下時）：**

- 送信先：申請者（`requester_user_id`）
- 却下通知タイトル：`「シフト変更却下」`
- 却下通知本文：`「申請されたシフト変更が却下されました。」` + 変更内容の要約 + 却下理由
- 却下通知の `metadata`：

```json
{
  "type": "shift_change_rejected",
  "request_id": "元の申請ID",
  "rejected_by": "事務部ユーザーID"
}
```

- `shift_change_requests` テーブルのステータスを `completed` または `rejected` に更新する。
- 同一申請への重複対応を防ぐため、`status !== 'pending'` チェックを行う。

### 機能 2：シフトリマインド通知

#### 2-1. 自動リマインド通知

**概要：**
全ユーザーに対し、シフト開始 20 分前にリマインド通知を送信する。

**詳細仕様：**

- GAS の時間ベーストリガーで定期実行する（10 分間隔）。
- 実行時、Google Sheets のシフトデータから現在時刻の 20 分後に開始するシフトを検出する。
- 該当するシフトの部員に対し、Supabase の `dispatch-notification` Edge Function を呼び出して通知を送信する。
- 通知タイトル：`「シフトリマインド」`
- 通知本文：`「20分後にシフトが始まります。{時間帯} - {エリア名}」`
- `metadata` に以下を格納する：

```json
{
  "type": "shift_reminder",
  "date": "2026-11-03",
  "time_slot": "10:00-10:30",
  "area_name": "ステッカー"
}
```

- 同一シフトに対する重複通知を防ぐため、送信済みフラグを管理する（GAS の PropertiesService を使用）。
- 通知送信にはユーザー名から `user_profiles` を逆引きして `user_id` を取得する必要がある。GAS から Supabase API を呼び出して取得する。

## 画面設計

### 1. 当日部員画面（タブ追加）

**画面名:** JimuShiftScreen（既存画面にタブを追加）
**パス/識別子:** 「当日部員」ドロワールート

**表示項目（変更点）：**

- タブバー：「マイシフト」|「シフト変更申請」|「変更申請管理」
  - 「マイシフト」タブは全ユーザーに表示（既存の個人シフト表示）
  - 「シフト変更申請」タブは `祭実長` または `部長` ロールのユーザーにのみ表示
  - 「変更申請管理」タブは `事務部` ロールのユーザーにのみ表示
  - タブバー自体は上記いずれかのロールを持つユーザーにのみ表示する

**動作：**

1. 画面表示時にユーザーのロールを確認
2. 該当ロールの場合はタブバーを表示
3. それ以外のユーザーは既存の画面をそのまま表示（タブバーなし）

### 2. シフト変更申請画面

**画面名:** ShiftChangeRequestScreen
**パス/識別子:** 「シフト変更申請」タブ内

**表示項目：**

- 日付選択（既存と同様のナビゲーション）
- 団体名表示
- シフトグリッド（行：メンバー名、列：時間帯 30 分刻み）
- 選択中セルのハイライト表示
- 操作ガイドテキスト（「交代してほしいシフトを選択してください」→「同じ時間帯の交代相手を選択してください」）
- 「選択をリセット」ボタン（交代元選択後に表示）

**レイアウト：**

- グリッドは横スクロール対応（時間帯が多いため）
- メンバー名列は固定表示（スクロール時も左端に固定）
- セルの色分け：シフトありセルは水色（`#4FC3F7`）、交代元選択中セルは青（`#1976D2`）、交代先選択中セルはオレンジ（`#FF9800`）、空きセルは白

**動作：**

1. 日付選択に応じて GAS API からシフトデータを取得
2. ユーザーの所属団体ブロックのみグリッドに表示
3. シフトが入っているセルをタップ → 「交代元」として選択（ハイライト）、他の時間帯列はグレーアウト
   - 連続する同一エリアの複数セルが存在する場合は**ブロック全体を自動選択**
4. 同じ時間帯の別メンバーのセルをタップ → 「交代先」として選択（ブロックサイズを自動補完）
5. 確認モーダルを表示
6. 「申請」ボタン押下で事務部に通知送信
7. 送信成功 → 成功メッセージ表示、選択状態をリセット
8. 送信失敗 → エラーメッセージ表示

---

### 3. シフト変更確認モーダル（申請者向け）

**画面名:** ShiftChangeRequestModal
**表示条件:** 移動元・移動先の両方が選択された時

**表示項目：**

- 変更日付
- 変更内容プレビュー（Before → After）
  - 移動パターン：「田中さん (10:00-10:30 ステッカー) → 山田さん (10:00-10:30 シフトなし)」
  - 交換パターン：「田中さん (10:00-10:30 ステッカー) ↔ 山田さん (10:00-10:30 渉外部B館前)」
- 備考入力欄（任意、最大 200 文字）
- 「申請」ボタン
- 「キャンセル」ボタン

**動作：**

1. 変更内容の確認を促す
2. 「申請」→ 通知送信処理を実行
3. 「キャンセル」→ モーダルを閉じ、交代先の選択状態のみリセット（交代元は保持）

**動作：**

### 5. シフト変更対応モーダル（事務部向け）

**画面名:** ShiftChangeResponseModal
**表示条件:** 申請カードタップ時

**表示項目：**

- 申請詳細（変更内容・申請者備考・ステータス）
- 対応備考入力欄（必須、最大 500 文字）
- 「承認」ボタン / 「却下」ボタン（未対応の申請のみ表示）
- インライン確認エリア（ボタン押下後に表示）
  - 確認メッセージ
  - 「戻る」ボタン
  - 「完了にする」または「却下する」ボタン
- エラーメッセージ（送信失敗時）
- 「閉じる」ボタン

**動作：**

1. 申請内容を表示
2. 対応備考を入力
3. 「承認」または「却下」を押す → インライン確認エリアを表示（`Alert.alert` は使用しない）
4. 「完了にする / 却下する」を押す → 申請者に通知送信 + ステータス更新
5. 成功 → モーダルを閉じ、一覧を再取得
6. 既に対応済みの申請はボタン非表示・対応備考を閲覧のみ可能

---

### 6. 通知一覧画面（事務部向け変更点）

**画面名:** NotificationListScreen（既存画面）

事務部は通知一覧でシフト変更申請通知を受け取るが、対応操作は**専用の「変更申請管理」タブ**（ShiftChangeRequestListScreen）で行う。

- 通知一覧に「承認」ボタンは表示しない（専用タブへ誘導する設計）。
- 通知をタップすることで申請の存在を把握し、「変更申請管理」タブで対応する。

**通知送信の認可：**

- アプリから `dispatch-notification` Edge Function を呼び出す際、`部長`・`祭実長`・`管理者` ロールを持つユーザーの Bearer トークンを認可する。

### shift_change_requests テーブル

**説明:** シフト変更申請の履歴と状態を管理

| カラム名 | 型 | 制約 | デフォルト | 説明 |
| --- | --- | --- | --- | --- |
| id | uuid | PK | gen_random_uuid() | 申請 ID |
| requester_user_id | uuid | NOT NULL, FK(auth.users.id) | - | 申請者ユーザー ID（祭実長・部長） |
| organization_name | text | NOT NULL | - | 対象団体名 |
| shift_date | date | NOT NULL | - | シフト日付 |
| source_member_name | text | NOT NULL | - | 移動元メンバー名 |
| source_time_slot | text | NOT NULL | - | 移動元時間帯（ブロック選択時は合成した範囲文字列、例：`"8:00-9:00"`） |
| source_area_name | text | NOT NULL | - | 移動元エリア名 |
| destination_member_name | text | NULL | - | 移動先メンバー名 |

**リレーション：**

- `shift_change_requests.requester_user_id` → `auth.users.id`
- `shift_change_requests.notification_id` → `notifications.id`

- `(requester_user_id, created_at DESC)` — 申請者別の履歴取得用
- `(status, created_at DESC)` — ステータス別の一覧取得用

### マイグレーション（作成用 SQL）

```sql
-- シフト変更申請テーブル
CREATE TABLE IF NOT EXISTS shift_change_requests (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  requester_user_id uuid NOT NULL REFERENCES auth.users(id),
  organization_name text NOT NULL,
  shift_date date NOT NULL,
  source_member_name text NOT NULL,
  source_time_slot text NOT NULL,
  source_area_name text NOT NULL,
  destination_member_name text,
  destination_time_slot text NOT NULL,
  destination_area_name text,
  requester_note text,
  responder_note text,
  status text NOT NULL DEFAULT 'pending'
    CHECK (status IN ('pending', 'completed', 'rejected')),
  notification_id uuid REFERENCES notifications(id),
  completed_by uuid REFERENCES auth.users(id),
  completed_at timestamptz,
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now()
);

COMMENT ON TABLE shift_change_requests IS 'シフト変更申請（祭実長・部長→事務部）';

-- インデックス
CREATE INDEX IF NOT EXISTS idx_shift_change_requests_requester
  ON shift_change_requests (requester_user_id, created_at DESC);

CREATE INDEX IF NOT EXISTS idx_shift_change_requests_status
  ON shift_change_requests (status, created_at DESC);

-- RLS 有効化
ALTER TABLE shift_change_requests ENABLE ROW LEVEL SECURITY;

-- RLS ポリシー
-- 認証済みユーザーは自分の申請を SELECT 可能
CREATE POLICY shift_change_requests_select_own
  ON shift_change_requests FOR SELECT
  USING (auth.uid() = requester_user_id);

-- 事務部ユーザーは全ての申請を SELECT 可能
CREATE POLICY shift_change_requests_select_jimu
  ON shift_change_requests FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM user_roles ur
      JOIN roles r ON r.id = ur.role_id
      WHERE ur.user_id = auth.uid() AND r.name = '事務部'
    )
  );

-- 祭実長・部長のみ INSERT 可能
CREATE POLICY shift_change_requests_insert
  ON shift_change_requests FOR INSERT
  WITH CHECK (
    auth.uid() = requester_user_id
    AND EXISTS (
      SELECT 1 FROM user_roles ur
      JOIN roles r ON r.id = ur.role_id
      WHERE ur.user_id = auth.uid() AND r.name IN ('祭実長', '部長')
    )
  );

-- 事務部のみ UPDATE 可能（ステータス更新用）
CREATE POLICY shift_change_requests_update_jimu
  ON shift_change_requests FOR UPDATE
  USING (
    EXISTS (
      SELECT 1 FROM user_roles ur
      JOIN roles r ON r.id = ur.role_id
      WHERE ur.user_id = auth.uid() AND r.name = '事務部'
    )
  );
```

## API 設計

### 1. シフト変更申請サービス（内部呼び出し）

**ファイル:** `src/features/jimu-shift/services/shiftChangeService.js`

**主な関数：**

- `insertShiftChangeRequest(params)` — 変更申請レコードを作成し、事務部に通知送信
- `selectShiftChangeRequestById(requestId)` — 申請詳細を取得
- `selectShiftChangeRequestsByUser(userId)` — ユーザーの申請履歴を取得
- `selectAllShiftChangeRequests()` — 全申請を取得（事務部向け）
- `completeShiftChangeRequest(requestId, completedByUserId, responderNote)` — 承認処理・申請者へ通知
- `rejectShiftChangeRequest(requestId, rejectedByUserId, responderNote)` — 却下処理・申請者へ通知

---

### 2. 通知送信（既存サービスの利用）

**ファイル:** `src/shared/services/notificationService.js`（既存）

**使用する関数：**

- `sendNotificationToRole(roleId, title, body, metadata, senderUserId)` — 事務部への申請通知
- `sendNotificationToUser(userId, title, body, metadata, senderUserId)` — 完了フィードバック通知

---

### 3. GAS リマインド通知スクリプト

**ファイル:** Google Apps Script 側に新規作成

**処理フロー：**

```
1. 時間ベーストリガー（10分間隔）で実行
2. 現在時刻 + 20分 の時間帯を計算
3. 当日のシフトシートからデータ取得
4. 該当時間帯にシフトが入っているメンバーを検出
5. PropertiesService で送信済みチェック（重複防止）
6. 未送信のメンバーに対して Supabase API 経由で通知送信
   - user_profiles テーブルから名前で user_id を逆引き
   - dispatch-notification Edge Function を呼び出し
7. 送信済みフラグを記録
```

**認証：**

- GAS から Supabase API を呼び出す際は `X-Internal-Notify-Token` ヘッダーで認可（既存の Edge Function 仕様に準拠）。

## 非機能要件

### パフォーマンス

- シフトグリッド表示は 3 秒以内を目標
- 変更申請通知の送信は 2 秒以内を目標
- リマインド通知は GAS トリガー間隔（10 分）以内に全件送信を完了

- リマインド通知は祭り期間中のみ稼働（期間外はトリガーを無効化）
- GAS トリガーの実行失敗時は次回トリガーで再検出される設計

## セキュリティ設計

### 認証・認可

- **シフトグリッド閲覧：** `祭実長` または `部長` ロールを持つユーザーのみ。自団体のデータのみ表示。
- **変更申請送信：** `祭実長` または `部長` ロールを持つユーザーのみ。RLS の INSERT ポリシーで制御。
- **完了フィードバック：** `事務部` ロールを持つユーザーのみ。RLS の UPDATE ポリシーで制御。
- **リマインド通知：** GAS からの API 呼び出しは `X-Internal-Notify-Token` で認可。

### データ保護

- **通信：** HTTPS のみ
- **API キー管理：** GAS 側の Supabase 接続情報は PropertiesService（スクリプトプロパティ）で管理
- **RLS：** `shift_change_requests` テーブルで有効化
- **個人情報：** シフトデータに含まれる氏名は団体内メンバーにのみ表示（RLS + フロント側の権限チェックで制御）
