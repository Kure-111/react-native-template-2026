# プロジェクト仕様書 — 迷子通知・検索機能（Item5）

## 機能概要

大学祭（生駒祭）において、任意のユーザーが迷子を発見した際に「実長」「渉外部」権限を持つユーザーへ通知を送り、迷子情報を登録・管理する機能。

### ユーザーロール別の操作

| ロール | できること |
|---|---|
| 任意のユーザー（全権限） | 迷子の新規登録・通知送信、申請履歴タブから自身の申請ステータス確認 |
| 実長 / 渉外部 | 迷子一覧の閲覧、ステータス変更（対応中・保留中・対応完了）、コメント記入 |
| 実長のみ | 上記に加えて、全データ削除 |

**備考:** 実長と渉外部は基本的に同じ権限を持つ。ただし**全データ削除は実長のみ**が可能。以降の仕様書で「管理ロール」と記載した場合は「実長 または 渉外部」を指す。

### 対象ロール情報

| ロール名 | role name | role ID |
|---|---|---|
| 実長 | `実長` | `0ed76a89-e49c-4389-b029-66140f0640ca` |
| 渉外部 | `渉外部` | `565c0702-b387-4a94-8a94-7fb50f3ffe49` |

### 通知先

- 迷子登録時に `実長` ロール・`渉外部` ロールを持つ**全ユーザー**にプッシュ通知を送信
- 通知には `src/shared/services/notificationService.js` の `sendNotificationToRoles()` を使用（複数ロール同時送信）

## 画面設計

### 画面構成

Item5Screen は**タブ切り替え方式**で以下の2つ（管理ロールは3つ）のタブを持つ。

| タブ名 | 表示対象 | 説明 |
|---|---|---|
| 迷子登録 | 全ユーザー | 迷子発見時の新規登録フォーム |
| 申請履歴 | 全ユーザー | 自分が登録した迷子のステータス確認 |
| 迷子管理 | 管理ロールのみ | 全迷子一覧・ステータス変更・コメント・全データ削除（実長のみ） |

---

### タブ1: 迷子登録（全ユーザー）

**入力フォーム項目:**

| 項目名 | 入力形式 | 必須 | 備考 |
|---|---|---|---|
| 年齢 | テキスト自由入力 | ○ | 「3歳」など自由記述 |
| 性別 | プルダウン選択 | ○ | 選択肢: `男` / `女` / `その他` |
| 特徴 | テキスト自由入力 | ○ | プレースホルダ: 「服装や所持品、身長など」 |
| 発見場所 | テキスト自由入力 | ○ | 「B館前」など自由記述 |
| 保護テント | プルダウン選択 | ○ | 選択肢: `西門前テント` / `B館前テント` / `人工芝グラウンド前テント` / `移動不可` |
| 迎えに来て欲しい場所 | テキスト自由入力 | ※条件付き必須 | **保護テントで「移動不可」を選択した場合のみ表示・必須** |
| 発見時刻 | 自動補完（現在時刻） | — | 画面に表示するが**編集不可** |

> **注:** 名前はセキュリティ上の理由により申請フォームからは入力しない。管理ロール（実長・渉外部）が「対応・編集」モーダルから後から登録できる。

**「移動不可」選択時の動作:**

- 保護テントのプルダウンで「移動不可」を選択すると、直下に「迎えに来て欲しい場所」のテキスト入力欄が動的に表示される
- この入力欄は必須入力（空の場合は登録不可）
- 迷子が泣きすぎて保護テントまで案内できない等の緊急性の高いケースを想定

**動作フロー:**

1. ユーザーがフォームに必須項目を入力
2. 「登録」ボタンを押下
3. **確認モーダル**を表示（入力内容の一覧を表示）
   - 「この内容で申請しますか？」
   - 申請後の**編集・削除はできない**旨の注意文を表示
   - 「移動不可」の場合は確認モーダル内で**赤色の緊急バッジ**を表示
4. 「申請する」→ Supabase に保存 → 管理ロール全員にプッシュ通知送信
5. 「キャンセル」→ モーダルを閉じてフォームに戻る
6. 送信成功時 → 成功メッセージを表示し、フォームをリセット

---

### タブ2: 申請履歴（全ユーザー）

- ログインユーザーが自分で登録した迷子情報の一覧を表示
- 各申請カードに表示する情報:
  - 特徴
  - 発見場所
  - 保護テント（「移動不可」の場合は迎え場所も表示）
  - 発見時刻
  - **ステータスバッジ**（色分け表示）
  - 「移動不可」の場合は**緊急バッジ**を表示

**ステータス一覧:**

| ステータス | 値 | バッジ色 | 説明 |
|---|---|---|---|
| 未対応 | `pending` | グレー | 登録直後の初期状態 |
| 対応中 | `in_progress` | 青 | 管理ロールが対応を開始した |
| 保留中 | `on_hold` | 黄 | 管理ロールが保留にした |
| 対応完了 | `completed` | 緑 | 管理ロールが対応を完了した |

- 最新のものが上に来るよう降順で表示
- 申請がない場合は「申請履歴がありません」を表示

---

### タブ3: 迷子管理（管理ロールのみ）

**表示条件:** ログインユーザーが `実長` または `渉外部` ロールを持つ場合のみタブを表示

#### 迷子一覧

- 全ユーザーが登録した迷子情報を一覧表示
- 各カードに表示する情報:
  - 迷子の名前（登録済みの場合のみ）・年齢・性別
  - **特徴**
  - 発見場所・保護テント
  - **「移動不可」の場合:** カード全体を赤系の枠線・背景で強調表示し、「緊急: 移動不可」バッジ + 迎え場所を目立つ形で表示
  - 発見時刻
  - 登録者：`(ロール名)氏名` 形式（user_profiles + user_roles から取得）
  - ステータスバッジ
  - 管理ロールコメント（あれば）
- フィルタリング: ステータスで絞り込み可能（全件 / 未対応 / 対応中 / 保留中 / 対応完了）
  - 未対応・対応中・保留中のフィルタボタン右上に件数を赤丸バッジで表示（0件は非表示）
- ソート: 発見時刻の降順（最新が上）
- **「移動不可」の未対応案件は常にリストの最上部に固定表示**（緊急性が高いため）

#### ステータス変更・情報編集

- 各カードにステータス変更ボタン（プルダウンまたはボタングループ）
- 変更可能なステータス: `対応中` / `保留中` / `対応完了`
- ステータス変更時に**任意テキスト入力のコメント欄**を表示
  - コメントは任意入力（空でもステータス変更可能）

**「移動不可」案件の情報編集（管理ロールのみ）:**

- `shelter_tent = 'unable_to_move'` の案件については、ステータス変更モーダル内に以下の編集フィールドを追加表示する
  - **保護テント:** プルダウン（通常の保護テント選択肢 + 移動不可）
    - 他のテントに変更した場合は `pickup_location` を非表示・クリアして保存
  - **迎えに来て欲しい場所:** テキスト入力（保護テントが「移動不可」のままの場合は必須）
- 編集はステータス変更と同一のモーダル内で完結する（別画面への遷移不要）
- ステータスは変更しなくても情報だけ更新できる（ステータスは現在値を維持）
- 更新対象カラム: `name`, `shelter_tent`, `pickup_location`, `status`, `admin_comment`, `updated_at`

**名前の登録（管理ロールのみ）:**

- 「対応・編集」モーダル内に「名前（任意）」入力フィールドを追加
- 申請フォームでは名前を収集しないため、管理ロールが迷子に確認できた際にここから登録する
- 入力は任意（空のまま更新も可能）
- モーダルのサマリー部分に: 名前（登録済みなら）・年齢/性別/特徴・発見場所・保護テント・登録者（`(ロール)氏名`形式）を表示

#### 全データ削除機能（実長のみ）

**表示条件:** ログインユーザーが `実長` ロールを持つ場合のみ表示（渉外部には非表示）

**配置場所:** 迷子管理タブ内の上部または下部に「全データ削除」ボタンを配置

**削除条件:**

| 条件 | 削除可否 | UI表現 |
|---|---|---|
| 祭期間中（`EXPO_PUBLIC_FESTIVAL_START_DATE` 〜 `EXPO_PUBLIC_FESTIVAL_END_DATE`） | 不可 | ボタンがグレーアウト。「生駒祭期間中のためこの操作は無効です。」を表示 |
| 祭期間外 かつ 未完了の迷子情報あり | 不可 | ボタンがグレーアウト。「全ての迷子情報が対応完了になっていません。」を表示 |
| 祭期間外 かつ 全件対応完了 | 可能 | ボタンが有効 |

**削除フロー（二重確認モーダル）:**

1. 「全データ削除」ボタンを押下
2. **1回目の確認モーダル:**
   - 「迷子データを全件削除します。この操作は取り消せません。」
   - 注意文: 「この操作は、生駒祭行事全てが完了した後に行なってください。」
   - 「削除する」/「キャンセル」
3. **2回目の確認モーダル:**
   - 「本当に削除しますか？ 全 XX 件の迷子データが完全に削除されます。」
   - 「はい、削除します」/「やめる」
4. 削除実行 → 成功メッセージ表示 → リスト更新

#### デバッグ用: 現在日付の操作

- 迷子管理タブ内に「デバッグ用: 現在日付の操作」セクションを配置
- 日付ピッカーまたはテキスト入力で任意の日付を設定可能
- 設定した日付で祭期間判定を上書きできる（全データ削除ボタンのグレーアウト確認用）
- 「リセット」ボタンで実際の現在日付に戻す
- **本番リリース時にはこのセクションを非表示にすること**（定数 `IS_DEBUG_MODE` で制御）

## データベース設計

### 新規テーブル: `missing_children`

**説明:** 迷子情報を管理するテーブル

| カラム名 | 型 | 制約 | 説明 |
|---|---|---|---|
| id | uuid | PK, DEFAULT gen_random_uuid() | レコードID |
| name | text | NOT NULL | 迷子の名前 |
| age | text | NOT NULL | 年齢（自由記述） |
| gender | text | NOT NULL | 性別（`male` / `female` / `other`） |
| characteristics | text | NOT NULL | 特徴（服装・所持品・身長など） |
| discovery_location | text | NOT NULL | 発見場所（自由記述） |
| shelter_tent | text | NOT NULL | 保護テント（`west_gate` / `b_building` / `artificial_turf` / `unable_to_move`） |
| pickup_location | text | | 迎えに来て欲しい場所（`shelter_tent = 'unable_to_move'` の場合のみ必須） |
| discovered_at | timestamptz | NOT NULL | 発見時刻 |
| status | text | NOT NULL, DEFAULT 'pending' | ステータス（`pending` / `in_progress` / `on_hold` / `completed`） |
| admin_comment | text | | 管理ロールのコメント |
| reported_by | uuid | NOT NULL, FK → auth.users(id) | 登録者のユーザーID |
| created_at | timestamptz | NOT NULL, DEFAULT now() | 作成日時 |
| updated_at | timestamptz | NOT NULL, DEFAULT now() | 更新日時 |

**CHECK制約:**

- `gender IN ('male', 'female', 'other')`
- `shelter_tent IN ('west_gate', 'b_building', 'artificial_turf', 'unable_to_move')`
- `status IN ('pending', 'in_progress', 'on_hold', 'completed')`
- **`shelter_tent = 'unable_to_move' の場合、pickup_location は NOT NULL` であること**（アプリ側バリデーションで担保。DBレベルではCHECK制約で実装）

**インデックス:**

- `reported_by` — 申請履歴タブで自分の申請を取得する際に使用
- `status` — ステータスでのフィルタリング用
- `discovered_at` — 時系列ソート用
- `shelter_tent` — 移動不可案件の優先表示用

**RLS ポリシー:**

| ポリシー名 | 操作 | 対象 | 条件 |
|---|---|---|---|
| `missing_children_select_own` | SELECT | 全認証ユーザー | `reported_by = auth.uid()` |
| `missing_children_select_admin` | SELECT | 管理ロール（実長 / 渉外部） | 全件閲覧可能 |
| `missing_children_insert` | INSERT | 全認証ユーザー | `reported_by = auth.uid()` |
| `missing_children_update_admin` | UPDATE | 管理ロール（実長 / 渉外部） | `status`, `admin_comment`, `shelter_tent`, `pickup_location`, `updated_at` を更新可能 |
| `missing_children_delete_jitcho` | DELETE | 実長のみ | 全件削除可能 |

**備考:** RLSでのロール判定は `user_roles` テーブルと `roles` テーブルをJOINし、`roles.name IN ('実長', '渉外部')` で判定する。

## 通知設計

### 通常の迷子登録時の通知（保護テントに案内できた場合）

| 項目 | 値 |
|---|---|
| 送信先 | `実長` + `渉外部` ロール全員 |
| 使用API | `sendNotificationToRoles(ADMIN_ROLE_IDS, title, body, metadata, senderUserId)` |
| タイトル | `【迷子通知】迷子が発見されました` |
| 本文 | `{発見場所}で迷子が発見されました。名前: {名前}、特徴: {特徴}、保護先: {保護テント名}` |
| metadata | `{ type: 'missing_child', missingChildId: '{id}', isUrgent: false }` |

### 緊急の迷子登録時の通知（移動不可の場合）

| 項目 | 値 |
|---|---|

## マイグレーション SQL

```sql
-- missing_children テーブル作成
CREATE TABLE public.missing_children (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  age text NOT NULL,
  gender text NOT NULL CHECK (gender IN ('male', 'female', 'other')),
  characteristics text NOT NULL,
  discovery_location text NOT NULL,
  shelter_tent text NOT NULL CHECK (shelter_tent IN ('west_gate', 'b_building', 'artificial_turf', 'unable_to_move')),
  pickup_location text,
  discovered_at timestamptz NOT NULL,
  status text NOT NULL DEFAULT 'pending' CHECK (status IN ('pending', 'in_progress', 'on_hold', 'completed')),
  admin_comment text,
  reported_by uuid NOT NULL REFERENCES auth.users(id),
  created_at timestamptz NOT NULL DEFAULT now(),
  updated_at timestamptz NOT NULL DEFAULT now(),
  -- 移動不可の場合は迎え場所が必須
  CONSTRAINT chk_pickup_location_required CHECK (
    shelter_tent != 'unable_to_move' OR pickup_location IS NOT NULL
  )
);

-- コメント
COMMENT ON TABLE public.missing_children IS '迷子情報管理テーブル（発見・保護・対応状況を記録）';
COMMENT ON COLUMN public.missing_children.characteristics IS '迷子の特徴（服装・所持品・身長など）';
COMMENT ON COLUMN public.missing_children.pickup_location IS '迎えに来て欲しい場所（移動不可時のみ使用）';

-- インデックス
CREATE INDEX idx_missing_children_reported_by ON public.missing_children(reported_by);
CREATE INDEX idx_missing_children_status ON public.missing_children(status);
CREATE INDEX idx_missing_children_discovered_at ON public.missing_children(discovered_at DESC);
CREATE INDEX idx_missing_children_shelter_tent ON public.missing_children(shelter_tent);

-- RLS有効化
ALTER TABLE public.missing_children ENABLE ROW LEVEL SECURITY;

-- RLSポリシー: 自分の申請を閲覧可能（全認証ユーザー）
CREATE POLICY missing_children_select_own ON public.missing_children
  FOR SELECT
  USING (reported_by = auth.uid());

-- RLSポリシー: 管理ロール（実長・渉外部）は全件閲覧可能
CREATE POLICY missing_children_select_admin ON public.missing_children
  FOR SELECT
  USING (
    EXISTS (
      SELECT 1 FROM public.user_roles ur
      JOIN public.roles r ON ur.role_id = r.id
      WHERE ur.user_id = auth.uid()
        AND r.name IN ('実長', '渉外部')
    )
  );

-- RLSポリシー: 全認証ユーザーが登録可能（自分のIDのみ）
CREATE POLICY missing_children_insert ON public.missing_children
  FOR INSERT
  WITH CHECK (reported_by = auth.uid());

-- RLSポリシー: 管理ロール（実長・渉外部）のみステータス・コメント更新可能
CREATE POLICY missing_children_update_admin ON public.missing_children
  FOR UPDATE
  USING (
    EXISTS (
      SELECT 1 FROM public.user_roles ur
      JOIN public.roles r ON ur.role_id = r.id
      WHERE ur.user_id = auth.uid()
        AND r.name IN ('実長', '渉外部')
    )
  );

-- RLSポリシー: 実長のみ削除可能（渉外部は不可）
CREATE POLICY missing_children_delete_jitcho ON public.missing_children
  FOR DELETE
  USING (
    EXISTS (
      SELECT 1 FROM public.user_roles ur
      JOIN public.roles r ON ur.role_id = r.id
      WHERE ur.user_id = auth.uid()
        AND r.name = '実長'
    )
  );
```

## 祭期間判定ロジック

```javascript
/**
 * 現在が祭期間内かどうかを判定する
 * @param {Date|null} debugDate - デバッグ用の日付（nullの場合は現在日付を使用）
 * @returns {boolean} 祭期間内の場合true
 */
const isFestivalPeriod = (debugDate = null) => {
  const now = debugDate || new Date();
  const currentYear = now.getFullYear();

  // .env の EXPO_PUBLIC_FESTIVAL_START_DATE, EXPO_PUBLIC_FESTIVAL_END_DATE を使用
  // 形式: "MM-DD"（例: "11-01"）
  const startStr = process.env.EXPO_PUBLIC_FESTIVAL_START_DATE; // "11-01"
  const endStr = process.env.EXPO_PUBLIC_FESTIVAL_END_DATE;     // "11-05"

  const [startMonth, startDay] = startStr.split('-').map(Number);
  const [endMonth, endDay] = endStr.split('-').map(Number);

  const startDate = new Date(currentYear, startMonth - 1, startDay, 0, 0, 0);
  const endDate = new Date(currentYear, endMonth - 1, endDay, 23, 59, 59);

  return now >= startDate && now <= endDate;
};
```

## 「移動不可」案件の緊急表示仕様

### カード表示（迷子管理タブ）

- カード枠線: `#F44336`（赤）、2px
- カード背景色: `#FFF3F3`（薄い赤）
- 「緊急: 移動不可」バッジを名前の横に表示（赤背景・白文字）
- 「迎え場所: {pickup_location}」を太字で表示
- 未対応 (`pending`) の移動不可案件はリスト最上部に固定

### カード表示（申請履歴タブ）

- 通常カードに「緊急」バッジを付与（赤背景・白文字）
- 「迎え場所: {pickup_location}」を表示

### ソートロジック（迷子管理タブ）

```
1. shelter_tent = 'unable_to_move' AND status = 'pending' → 最上位（発見時刻降順）
2. それ以外 → 発見時刻降順
```
