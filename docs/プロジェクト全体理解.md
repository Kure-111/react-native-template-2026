## 1. このアプリの正体

- 名称: 生駒祭 ERP 2026（大学祭運営向け）
- 実装形態: React Native (Expo) を Web/モバイル両対応で運用
- バックエンド: Supabase（DB/Auth/Storage/Realtime/Edge Functions）
- 現在の開発中心: 管理部統合システム（`item12`〜`item16`）

---

## 2. 起動方法（最短）

```bash
npm install
cp .env.example .env
npm run web
```

補助スクリプト:
- `./START_SERVER.sh`: Web + 地震監視サービスを同時起動
- `./START_EARTHQUAKE_MONITOR.sh`: 地震監視のみ起動

---

## 3. 必須環境変数（主要）

- `EXPO_PUBLIC_SUPABASE_URL`
- `EXPO_PUBLIC_SUPABASE_ANON_KEY`
- `EXPO_PUBLIC_VAPID_PUBLIC_KEY`
- `WEB_PUSH_VAPID_PUBLIC_KEY`
- `WEB_PUSH_VAPID_PRIVATE_KEY`
- `WEB_PUSH_VAPID_SUBJECT`
- `INTERNAL_NOTIFY_TOKEN`

---

## 4. 主要ディレクトリ

- `src/features/`: 機能単位画面
- `src/navigation/`: Drawer/画面遷移
- `src/services/supabase/`: ドメインサービス（チケット、巡回、鍵など）
- `src/shared/`: 共通UI、通知、テーマ、認証
- `db/migrations/`: 適用SQL（通知、鍵、巡回関連テーブルなど）
- `supabase/functions/`: Edge Functions（通知配信、push購読）
- `docs/`: 仕様書・運用文書

---

## 5. ロール別の主画面（現在の実装）

- `item12`: 巡回サポート（タスク受諾/完了、巡回チェック、未巡回アラート、評価入力）
- `item13`: 本部サポート（全連絡案件対応、ダッシュボード、巡回割当、評価承認、無線、鍵管理）
- `item14`: 会計対応（対象案件の回答/状態更新）
- `item15`: 物品対応（対象案件の回答/状態更新）
- `item16`: 企画者サポート（質問、緊急、鍵事前申請、開始/終了報告）

補足:
- 権限制御は `roles.permissions.screens` による画面可視制御（`permissionService.js`）。

---

## 6. バックエンドの中心モデル

主テーブル:
- `support_tickets`: 連絡案件
- `ticket_messages`: 返信スレッド
- `patrol_tasks`: 巡回タスク
- `patrol_task_results`: 巡回結果
- `key_loans`, `key_reservations`, `keys`: 鍵運用
- `evaluation_checks`: 評価承認フロー
- `radio_logs`: 無線ログ
- `notifications`, `push_subscriptions`: 通知

主要RPC:
- `rpc_create_ticket_and_auto_tasks`
- `rpc_accept_task`
- `rpc_complete_task`
- `rpc_return_key_and_create_lock_task`

---

## 7. 通知と外部連携

- 連絡案件・巡回イベント通知:
  - `src/shared/services/supportWorkflowNotificationService.js`
  - `supabase/functions/dispatch-notification/index.ts`
- Web Push購読管理:
  - `supabase/functions/push-subscription/index.ts`
- 地震監視:
  - `src/services/earthquakeMonitor.js`
  - `docs/earthquake-monitor-service.md`

---

## 8. 現時点で重要な実装メモ

- 実装は `StyleSheet` ベースが中心（READMEの一部記述と差異あり）。
- 設置型/持ち歩きは「同一Expoアプリ内でロール別画面を切替える」構成。
- `item16` 送信処理では `eventId` が現状 `null` 固定のため、仕様書の「開始/終了報告時の `event_id` 必須」と差分が残る。
