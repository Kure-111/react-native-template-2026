## プロジェクト概要

### 目的

臨時ヘルプ（一時的に人手が足りない部署に対して人材を派遣する）システムに、各部署の部長のみが利用できる募集機能を提供する。

### スケジュール

- 開発開始：2026/02/09
- リリース予定：未定

---

## 対象ユーザー

### 各部署の部長

- **いつ使うか：** 人手が不足したタイミング
- **どういうときに使うか：** 部署の募集枠を作成・公開したいとき

### 各部署の一般社員

- **いつ使うか：** 募集の閲覧・応募を検討するとき
- **どういうときに使うか：** 募集を確認するが、募集作成はできない

## 機能要件

### 1. 臨時ヘルプタブの初期表示

**概要：**
ユーザーの権限に応じて表示内容を出し分ける。

**詳細仕様：**

- 部長（manager）の場合：臨時ヘルプタブを開くと募集作成を主領域に表示し、画面端に現在の募集中案件一覧（サマリ）を常時表示する
- 一般ユーザー（member）の場合：臨時ヘルプタブを開くと直接応募タブ（募集一覧）を表示し、「募集」ボタンは表示しない

---

### 2. 募集機能の権限確認

**概要：**
募集機能は部長権限があるユーザーのみ利用できる。

**詳細仕様：**

- 「募集」選択時にユーザーの権限を確認する
- 部長権限が確認できた場合のみ募集機能を利用可能
- 権限がない場合は募集機能に遷移させず、権限不足のメッセージを表示する

---

### 3. 募集作成（部長のみ）

**概要：**
部長が部署の募集を作成できる。

**詳細仕様：**

- 募集の作成は部長権限が必須
- 必須項目は「募集人数」「場所」「業務内容」「報酬」の4点  
  - 任意項目：「集合時間」「集合場所」「持ち物」  
    - 集合時間が未入力の場合は表示しない  
    - 集合場所が未入力の場合は「場所」と同じ値で表示する  
    - 持ち物が未入力の場合は「なし」と表示する
- 必須項目を基に募集を登録し、応募タブの一覧から確認できる
- 作成者（部長）のユーザー ID を `head_user_id` として記録する。後続の編集権限判定に利用
- 募集部署は作成者の所属部署を自動設定することを基本とし、必要に応じて作成時に部署を手動選択できるオプションを用意
- 募集人数・場所・業務内容の文字数上限は設けない（入力者の信頼性を前提にする）
- 将来的に必須項目が追加される可能性があるため、バリデーションは追加項目に柔軟に対応できる設計とする
- 作成後、募集一覧に反映される

---

### 4. 応募タブでの募集一覧表示

**概要：**
応募タブから作成済みの募集案件を閲覧できる。

**詳細仕様：**

- 一般社員を含む全ユーザーが応募タブで募集一覧を確認できる
- 部署をまたいで全募集を閲覧できる（閲覧制限なし）
- 一覧表示項目：募集人数、場所、集合場所（未入力時は場所と同じ値）、集合時間（入力時のみ表示）、業務内容、持ち物（未入力時は「なし」）、作成者名（または部署）、募集ステータス、作成日時
- 並び順は更新日時の降順（最新が上）をデフォルトとし、募集人数順・作成日時順の並び替えをサポート  
  - `closed` を再開した際は `updated_at` を現在時刻に更新し、一覧先頭に表示される
- 検索・絞り込み：場所／部署のフィルタを最低限サポート（拡張余地あり）
- ステータスは `open` / `closed` の2値想定（将来拡張可）
- `closed` の募集は一覧には表示しない（バックエンドでは保持）

---

### 5. 募集編集（作成者のみ）

**概要：**
作成者が自分の募集のみ編集できる。

**詳細仕様：**

- 編集操作は作成者（部長）のみ許可する。権限チェックは API とフロントの両方で実施
- 編集可能項目：募集人数、場所、業務内容（必須3項目）に加え、集合時間・集合場所・持ち物（任意）。将来追加項目は後方互換のまま同様に対応
- 編集対象の `head_user_id` と現在のユーザー ID が一致しない場合は 403（Forbidden）を返す
- 他者が編集しようとした場合は 403（Forbidden）を返し、フロントでエラーメッセージを表示
- 編集成功後は応募タブの一覧へ戻し、更新内容を反映

---

### 6. 募集公開期間とクローズ条件

**概要：**
募集の公開期間を自動/手動で終了させ、ステータスを `closed` にする。

**詳細仕様：**

- 以下いずれかで `closed` へ遷移する  
  - 応募数が募集人数に到達した時（定員到達）  
  - 募集作成日の翌日 00:00（同日限りの有効期間）  
  - 募集作成者が「募集終了」を手動選択した時
- クローズ後は応募不可。応募タブ一覧ではステータス `closed` で表示し、新規応募ボタンを非表示
- 自動クローズはバックエンド（Supabase スケジューラ/Edge Functions 等）で日次チェックする
- 作成者は `closed` の募集を再度 `open` に戻せる（手動再開）。再開時は `status=open` とし、`updated_at` を現在時刻に更新して一覧の先頭に並ぶようにする

---

### 7. 応募フロー

**概要：**
応募タブから案件を選択して応募する。

**詳細仕様：**

- 応募タブに表示される一覧から案件を選択し、応募ボタンで応募を登録
- 応募時に応募者のユーザー ID を紐付けて保存（Supabaseで識別可能前提）
- 募集作成者は応募者一覧を確認できるが、一般ユーザーには表示しない
- `closed` の募集は応募ボタンを無効化

---

### 8. 通知

**概要：**
募集・応募に応じた通知を行う。

**詳細仕様：**

- 募集作成時：全ユーザーへ新規募集の通知
- 募集編集時：その募集に応募済みのユーザーへのみ通知
- 応募発生時：募集作成者へ通知
- 通知手段（プッシュ/メール/社内チャット）は未定。手段が決まり次第、送信チャネルとレート制限を定義する

| updated_at | timestamptz | NOT NULL | 更新日時 |

**リレーション:**

- rinji_help_recruits.department_id → departments.id
- rinji_help_recruits.head_user_id → auth.users.id（user_profiles.user_id と一致）
- head_user_id から `user_roles`/`roles` を参照し、`manager` 権限を確認

**インデックス・制約:**

- 作成者別の編集判定用に `(id, head_user_id)` の複合インデックスを付与
- ステータス絞り込みと新着順のため `(status, updated_at DESC)` インデックスを付与（再開時に updated_at を更新して先頭に表示）

---

### rinji_help_applications テーブル

**説明:** 応募情報を管理

| カラム名 | 型 | 制約 | 説明 |
| ---------- | ----------- | ---------------- | ----------- |
| id | uuid | PK | 応募 ID |
| recruit_id | uuid | FK | 対象募集 ID |
| applicant_user_id | uuid | FK | 応募者ユーザー ID |
| status | text | NOT NULL, DEFAULT 'applied' | 応募ステータス（将来拡張用） |
| created_at | timestamptz | NOT NULL | 応募日時 |
| updated_at | timestamptz | NOT NULL | 更新日時 |

**リレーション:**

- rinji_help_applications.recruit_id → rinji_help_recruits.id
- rinji_help_applications.applicant_user_id → auth.users.id（user_profiles.user_id と一致）

**インデックス・制約:**

- 同一募集への重複応募を防ぐため `(recruit_id, applicant_user_id)` に UNIQUE 制約
- 募集別の応募者一覧取得用に `(recruit_id, created_at DESC)` インデックス

## セキュリティ設計

### 認証・認可

- **認証方式:** Supabase Auth（JWT）
- **認可:** `roles.name = 'manager'` を持つユーザーのみ募集作成/編集/クローズ/再開可、応募はログイン済み全ユーザー可
- **RLS ポリシー（基盤テーブルは概要ドキュメントに準拠）:**  
  - `user_profiles`: 自分の行のみ SELECT/UPDATE/INSERT 可（`auth.uid() = user_id`）。  
  - `user_roles`: 自分の行のみ SELECT 可（`auth.uid() = user_id`）。  
  - `roles`: 認証済みユーザーは SELECT 可。  
  - `rinji_help_recruits`: manager 権限を持つユーザーのみ `insert/update/delete`。`select` は全ユーザーが `status='open'` の行を取得可。  
  - `rinji_help_applications`: `insert` はログインユーザー自身の `applicant_user_id` のみ、`select` は作成者または応募者本人、`update` は無し（必要時に限定実装）。

### データ保護

- **通信:** HTTPS のみ
- **API キー管理:** 環境変数で管理
- **監査ログ/編集履歴:** 収集は行うが一般ユーザーへの開示は行わない。作成者・管理者のみが参照可能（UI 提供は今後検討）
