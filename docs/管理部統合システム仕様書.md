## 1. プロジェクト概要

### 1.1 目的

大学祭当日の「問い合わせ・緊急連絡・企画開始/終了報告・鍵・施錠確認・巡回・無線対応」を一元管理し、対応漏れ・伝達漏れ・属人化を減らし、状況を可視化する。

特に、企画管理部の当日運用を以下の観点で改善する。

- 受付（誰から・何が・いつ発生したか）を残す
- 対応（誰が・いつ・どう対応したか）を残す
- 依頼（巡回へ向かわせる／部署へ確認させる）をタスク化する
- 遅延（経過時間が長いもの）を見える化する

### 1.2 重要コンセプト（データモデルの骨格）

本システムは「連絡チケット」と「巡回タスク」の2本柱で回す。

- 連絡チケット（Ticket）
  - 出展団体からの連絡（緊急／質問／変更連絡／破損連絡 等）
  - 出展団体による企画開始報告・企画終了報告
  - 本部/会計/物品の回答（返信）をスレッドで保持

- 巡回タスク（Task）
  - 企画開始/終了報告を受けた「確認タスク」（巡回が向かう→結果を返す）
  - 鍵返却を受けた「施錠確認タスク」（巡回が向かう→施錠結果を返す）
  - 本部が作る「緊急対応」「定常巡回」等のタスク

### 1.3 端末区分（設置型／持ち歩き）

本プロジェクトは「設置型（本部固定）」と「持ち歩き（個人端末）」でUIを明確に分ける。

- 設置型（固定設置：本部サポート）
  - 本部に置くPC／固定タブレット用のWebダッシュボード
  - 大画面前提で「一覧・経過時間・同時監視・割当」が中心

- 持ち歩き（個人端末：出展／巡回／会計／物品）
  - スマホ前提（必要に応じてPCブラウザも可だが「固定設置」はしない）
  - 「入力・確認・返信・向かう/完了」中心
  - 同一アプリ（Expo）でロールにより表示を切替える想定

### 1.4 スコープ

対象：大学祭実行委員会 企画管理部の当日オペレーション全般  
会計部・物品部：通知の受信者に留めず、通知対象チケットに対して「閲覧・返信（回答）・対応メモ・完了」が可能な画面を提供する。

## 2. 対象ユーザー

### 2.1 企画管理部（本部）【設置型：固定PC/固定タブレット】

- できること
  - 全体状況の監視（新着、遅延、鍵、巡回、施錠確認、無線ログ）
  - チケットの受付・回答・状態更新・担当割当
  - 巡回タスクの作成・割当・進捗確認
  - 鍵管理（貸出／返却／予約）＋返却起点の施錠確認依頼
  - 評価承認
  - 部署（会計/物品）対応状況の追跡、必要時の再通知

### 2.2 企画管理部（巡回）【持ち歩き：スマホ】

- できること
  - タスク一覧の確認（未割当タスクも拾える）
  - 「向かいます」→「完了」＋結果入力
  - 施錠確認（鍵返却起点のタスク）
  - 企画開始/終了確認（出展報告起点のタスク）
  - 巡回チェック（定常巡回ログ）
  - 企画評価入力（承認待ち）

### 2.3 出展団体（企画責任者/当日担当）【持ち歩き：スマホ】

- できること
  - 緊急連絡、質問、配置図変更相談
  - 商品配布ルール変更連絡（会計へ通知）
  - 物品破損連絡（物品へ通知）
  - 鍵の事前申請（予約）
  - 企画開始報告／企画終了報告（報告後、巡回・本部で確認できる／巡回は向かう・完了可能）
  - 返信確認・追記（本部/会計/物品の回答が見える）

### 2.4 会計部【持ち歩き：スマホ（必要ならPCブラウザも可）】

- できること
  - 配布ルール変更連絡の通知受信
  - 内容確認、返信（回答）、対応メモ、完了

### 2.5 物品部【持ち歩き：スマホ（必要ならPCブラウザも可）】

- できること
  - 物品破損連絡の通知受信
  - 内容確認、返信（回答）、対応メモ、完了

### 3.1 連絡チケット（共通仕様：分かりやすい名称）

#### 3.1.1 チケットの基本項目

- 受付番号（自動採番）
- 種別（ticket_type）
- 状態（ticket_status）
- 優先度（priority）
- 発生場所（location）
- 対象企画（event）※企画開始/終了は必須
- 作成者（誰が出したか）
- 担当（本部担当：任意）
- 返信スレッド（本部/会計/物品/出展が投稿）
- 添付（写真/ファイル：任意）

#### 3.1.2 チケット状態（ticket_status）

- new：新規
- acknowledged：受付済（見た・受けた）
- in_progress：対応中（担当が動いている）
- waiting_external：他部署/確認待ち（会計/物品/巡回など）
- resolved：解決（対応完了）
- closed：クローズ（履歴固定）

#### 3.1.3 優先度（priority）

- high / normal / low

---

### 3.2 出展団体向け（当日サポート：持ち歩き）

#### 3.2.1 緊急連絡

- 必須：詳細、場所、優先度（基本high）
- 任意：写真
- 作成後：本部に即時表示

#### 3.2.2 ルール問い合わせ

- 必須：カテゴリ、本文、場所（基本）
- 任意：FAQ提示（自己解決導線）

#### 3.2.3 配置図変更の相談

- 必須：変更内容（Before/Afterが分かる）
- 任意：配置図添付
- 本部の承認結果（承認/差戻し/却下）を受信

#### 3.2.4 商品配布ルール変更の連絡（会計へ通知）

- 必須：変更内容、適用開始時刻、影響範囲
- 作成時：会計部へ自動通知
- 会計部の返信（回答）を出展団体が閲覧できる
- 本部も追跡できる（必要なら追加で本部が回答）

#### 3.2.5 物品破損の連絡（物品へ通知）

- 必須：破損物品名、状況、保管場所/現状、緊急度
- 推奨：写真（特に破損状況）
- 作成時：物品部へ自動通知
- 物品部の返信（回答）を出展団体が閲覧できる
- 本部も追跡できる

#### 3.2.6 鍵の事前申請（鍵予約）

- 必須：対象教室/鍵、希望時刻、理由、団体名
- 本部承認後：予約確定通知

#### 3.2.7 企画開始報告／企画終了報告（重要：巡回・本部で見える＋巡回が向かう/完了）

- 出展団体が「開始」「終了」を報告できる（別UIとして用意）
- 必須：対象企画（event_id）
- 任意：メモ、写真

報告後の動き（自動）：

1. 本部（設置型）に「開始/終了報告」が新着表示される
2. 巡回（持ち歩き）に「確認タスク」が発生し一覧に出る
3. 巡回はタスクで「向かいます」→現地確認→「完了」し、結果を本部に返す
4. 本部は結果を見て「対応完了」へ進められる（運用ルールで自動/手動を選択）

---

### 3.3 巡回向け（巡回サポート：持ち歩き）

#### 3.3.1 タスク一覧（確認一覧）

対象タスク例：

- 企画開始確認（出展開始報告起点）
- 企画終了確認（出展終了報告起点）
- 施錠確認（鍵返却起点）
- 緊急対応（本部発行）
- 定常巡回（本部発行／ルート巡回）

表示項目：

- 種別、場所、起点（報告/返却/指示）、経過時間、状態、担当

#### 3.3.2 タスクの進捗操作（向かいます／完了）

- 向かいます
  - status=open → accepted
  - accepted_at記録、担当者確定
- 完了
  - status=accepted/en_route → done
  - done_at記録、結果入力送信

状態（task_status）：

- open / accepted / en_route / done / canceled

#### 3.3.3 企画開始/終了確認（完了時入力）

- 結果：
  - 問題なし
  - 開始していない（開始確認の場合）
  - 終了していない（終了確認の場合）
  - 別対応必要（例：体調不良、トラブル）
- メモ、写真（任意）

#### 3.3.4 施錠確認（鍵返却起点）

- 本部が鍵返却を処理したら、施錠確認タスクが生成され巡回に出る
- 結果：
  - 施錠済
  - 未施錠
  - 確認不可
- 未施錠の場合は写真推奨（証跡）

#### 3.3.5 巡回チェック（定常巡回ログ）

入力：

- 場所
- 体調不良、困りごと、来場者トラブル、無人未施錠、その他
- メモ、写真（任意）

#### 3.3.6 未巡回アラート

- 場所ごとの最終巡回時刻を可視化
- 閾値（例：60分）以上でアラート（本部が設定可能）

#### 3.3.7 企画評価（承認フロー）

- 巡回が評価入力 → 承認待ち
- 本部が承認/差戻し

---

### 3.4 本部向け（本部サポート：設置型）

#### 3.4.1 本部ダッシュボード（全体監視）

表示ブロック：

- 新着チケット（緊急、問い合わせ、破損、配布変更、開始/終了報告）
- 遅延（経過時間が長いチケット/タスク）
- 施錠確認（未完了の施錠タスク）
- 鍵状況（貸出中、返却待ち、予約）
- 巡回状況（担当別 未対応数）
- 無線ログ（直近）

#### 3.4.2 チケット管理（一覧／詳細）

- 一覧：種別、優先度、状態、場所、経過時間、担当
- 詳細：
  - 返信（本部が回答）
  - 状態更新、担当割当
  - 会計/物品が関与する場合：部署の返信・完了状況を確認
  - 開始/終了報告の場合：紐づく「確認タスク」の進捗と結果を表示

#### 3.4.3 巡回タスク管理（一覧／詳細）

- タスクの割当（特定巡回へ割当／未割当運用も可）
- 進捗監視（向かいます/完了）
- 結果確認

#### 3.4.4 鍵管理（貸出／返却／予約）＋施錠確認依頼

- 鍵マスタ
- 予約申請（承認/差戻し）
- 貸出登録（借受人・団体・時刻）
- 返却登録（返却時刻・担当）

施錠確認の流れ（必須要件）：

- 返却処理時に「施錠確認を巡回へ依頼」を実行できる
- 実行すると lock_check タスクを生成し巡回へ送信
- 巡回結果（施錠済/未施錠/確認不可）を本部で確認する

#### 3.4.5 評価承認

- 承認待ち一覧
- 承認/差戻し（理由）

#### 3.4.6 無線ログ

- 記録主体：本部／巡回
- 重大度：S/A/B/C 等
- 検索/フィルタ（重大度、場所、時刻）

#### 3.4.7 部署連携（会計/物品）

- 自動通知ログ
- 部署返信・完了状況
- 再通知（手動）

- 詳細：

  - 対応メモ
  - 完了

### 4.1 出展（持ち歩き：スマホ）

#### 4.1.1 ExhibitorHome（受付一覧）

- 表示：自団体チケット一覧（状態、種別、経過時間、最終更新）
- 操作：新規作成、詳細へ遷移、フィルタ（未対応/対応中/完了）

- 種別選択（緊急／質問／配置図／配布変更／破損）

- 送信後：受付番号表示 → TicketDetailへ

#### 4.1.3 TicketDetail（詳細・返信）

- 表示：内容、状態、返信スレッド、添付
- 操作：追記投稿、添付追加（必要なら）

#### 4.1.4 EventStatus（企画開始/終了報告）

- 表示：対象企画、現在状態（未開始/実施中/終了）、履歴
- 操作：
  - 開始報告（start_report）送信
  - 終了報告（end_report）送信

---

### 4.2 巡回（持ち歩き：スマホ）

#### 4.2.1 PatrolTaskList（タスク一覧）

- 表示：自分のタスク＋未割当タスク（種別、場所、経過時間）

#### 4.2.2 PatrolTaskDetail（向かいます／完了）

- 表示：指示、起点、場所、状態、経過時間

#### 4.2.4 UnvisitedList（未巡回アラート）

- 表示：場所、最終巡回時刻、経過時間

#### 4.3.3 TaskList_HQ / TaskDetail_HQ

#### 4.3.4 KeyManagement_HQ（鍵＋施錠依頼）

  - 返却確定

## 6. API 設計

### 6.1 方針

- Supabase REST + Realtime を基本
- 整合性が重要な処理は RPC（Postgres Function）で原子的に処理
- 通知は Edge Function で一元送信し、必ず notifications にログを残す
- 企画開始/終了報告 → 巡回タスク自動生成
- 鍵返却 → 施錠確認タスク生成（返却画面の操作で実行）

### 6.2 推奨RPC（最小セット）

1. `rpc_create_ticket_and_auto_tasks(ticket_payload)`

- start_report/end_report の場合：confirm_start/confirm_end タスクを自動生成

2. `rpc_return_key_and_create_lock_task(loan_id, create_lock_task, optional_assignee)`

- 返却処理を確定
- create_lock_task=true の場合：lock_check タスクを生成

3. `rpc_accept_task(task_id, patrol_user_id)`

- open → accepted（同時に担当を確定、取り合い防止）

4. `rpc_complete_task(task_id, result_payload)`

- done_at 記録
- patrol_task_results 作成
- 必要なら本部側の通知を起動

### 6.3 通知（Edge Function）

- `functions/v1/notify`
  - 入力：event_type, target_role/org, payload
  - 出力：notificationsにログ、外部通知（Discord/LINE等）

通知トリガ（必須）：

- distribution_change → 会計部へ
- damage_report → 物品部へ
- （任意）緊急連絡 → 本部へ（外部通知も）

## 7. 非機能要件

### 7.1 パフォーマンス

- 本部ダッシュボード：主要一覧 3秒以内
- チケット作成〜本部反映：5秒以内（Realtime前提）

### 7.2 可用性

- 当日稼働最優先
- 持ち歩き側：通信不安定時の下書き保存（最低限：本文/種別/場所）

### 7.3 運用性（不足しがちな追加要件）

- マスタ管理
  - locations / events / keys / organizations の投入・更新手順（本番前に整備）
- 監査性
  - ticket_status / task_status の変更履歴を残す（最低限updated_at＋担当者、理想は監査ログテーブル）

### 7.4 対応環境

- 持ち歩き：iOS 15+ / Android 10+
- 設置型：Chrome最新版（PC/固定タブレット）

## 9. セキュリティ設計

### 9.1 認証

- Supabase Auth（JWT）

### 9.2 認可（RBAC）

- roles / user_roles によるロール制御
- user_organizations で団体境界を切る（出展の隔離）

### 9.3 RLS方針（要点）

- 出展（Exhibitor）
  - 自団体（org_id）の ticket だけ閲覧/作成/追記可
- 本部（HQ）
  - 全件閲覧/更新/割当可
- 会計（Accounting）
  - notify_target=accounting のチケット閲覧、返信投稿、完了更新可（範囲は運用で固定）
- 物品（Property）
  - notify_target=property のチケット閲覧、返信投稿、完了更新可
- 巡回（Patrol）
  - タスク閲覧（未割当＋自分担当）、accept/complete 可

### 9.4 添付（Storage）

- バケット分離（ticket_attachments）
- 期限付きURL or サイン付きURL
- RLSでファイル参照の権限制御（最低限：チケット閲覧可能者のみ）
