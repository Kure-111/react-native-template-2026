## 1. 最初に押さえること

- このリポジトリは `生駒祭 ERP 2026` の Expo アプリで、大学祭運営向けの業務システムです。
- 現在ここで主に開発しているのは `企画管理部統合システム` です。
- 通常の編集対象は `item12` から `item16`、およびそれに直結する `support` / `src/services/supabase` / 関連ドキュメントです。
- 明示的な依頼がない限り、`item1` から `item10` やその他の独立機能は触らない前提で進めます。
- 現在の実作業では、`docs/管理部統合システム仕様書.md` を最重要仕様として扱います。
- `docs/プロジェクト仕様書.md` にはテンプレート由来の記述も残っているため、管理部統合システムの実装判断では `docs/管理部統合システム仕様書.md` と実コードを優先して整合を見る必要があります。

## 2. 技術スタック

- フロントエンド: React Native + Expo SDK 54
- 実行対象: Web / iOS / Android
- 言語: JavaScript
- バックエンド: Supabase
- 通知: Supabase Edge Functions + Web Push
- 外部連携: Google Apps Script、地震監視サービス
- スタイリング: `StyleSheet` のみ

補足:
- `package.json` では `typescript` 依存が入っていますが、アプリ本体の実装方針は JavaScript です。
- TypeScript は `supabase/functions/` の Edge Functions 側のみ例外的に存在します。

## 4. アプリの全体構造

エントリーフロー:

```text
index.js
  -> App.js
    -> AuthProvider
      -> ThemeProvider
        -> TerminalProvider
          -> FontLoaderProvider
            -> AppNavigator
```

画面遷移の大枠:

```text
AppNavigator
  -> 未認証: LoginScreen
  -> 認証済み: DrawerNavigator
  -> 初回ログイン: PasswordChangeModal / PasswordChangeForm / PasswordSuccessModal
```

Drawer の実装前提:

- `src/navigation/DrawerNavigator.jsx` が全体の主要画面を管理します。
- Web では `768px` を境に、PC は常設サイドバー、モバイルはハンバーガーメニューです。
- 各画面は `ScreenErrorBoundary` でラップされています。

## 7. 企画管理部統合システムの中核モデル

このシステムは主に次の 2 本柱で動いています。

### 7.1 連絡案件

- テーブル: `support_tickets`
- 返信: `ticket_messages`
- 添付: `ticket_attachments`

扱う内容:

- 企画ルール変更相談
- 配置図変更相談
- 会計向け配布基準変更
- 物品破損報告
- 緊急呼び出し
- 鍵の事前申請
- 企画開始報告
- 企画終了報告

### 7.2 巡回タスク

- テーブル: `patrol_tasks`
- 結果: `patrol_task_results`
- 巡回チェック: `patrol_checks`

扱う内容:

- 企画開始確認
- 企画終了確認
- 施錠確認
- 緊急対応
- 定常巡回

## 8. 実装されている主要フロー

### 8.1 企画者が連絡する

1. `item16` で入力する
2. `exhibitorSupportService` が `support_tickets` を作成する
3. 必要に応じて添付を `ticket_attachments` に保存する
4. `supportWorkflowNotificationService` 経由で通知する
5. 種別によって本部 / 会計 / 物品へ振り分けられる

### 8.2 本部 / 会計 / 物品が対応する

1. `item13` / `item14` / `item15` が `SupportDeskScreen` を使って案件一覧を表示する
2. `supportTicketService` で案件一覧、返信、状態更新を行う
3. 本部は追加で巡回割当、評価承認、鍵管理、無線ログ登録を行う

### 8.3 巡回が対応する

1. `item12` で `patrol_tasks` を取得する
2. `acceptPatrolTask` で「向かいます」を処理する
3. `completePatrolTask` で結果を `patrol_task_results` に記録する
4. 必要に応じて元の連絡案件へメッセージや状態を反映する

### 8.4 鍵返却から施錠確認へつなぐ

1. 本部が `key_loans` の返却処理を行う
2. `returnKeyAndCreateLockTask` が動く
3. `rpc_return_key_and_create_lock_task` が使えれば RPC を優先する
4. 失敗時はアプリ側フォールバックで `patrol_tasks` に施錠確認タスクを作る

## 10. 認証と権限制御

- `AuthContext` がセッション、プロフィール、ロール情報を保持します。
- `user_profiles` と `user_roles -> roles` を統合して `userInfo` を作ります。
- 画面の可視制御は `roles.permissions.screens` を `permissionService.js` が参照します。
- 初回ログイン時はパスワード変更導線が出ます。
- ログイン後に Web Push 購読初期化も動きます。

## 12. 重要な RPC / 自動化ポイント

優先利用される RPC:

- `rpc_create_ticket_and_auto_tasks`
- `rpc_accept_task`
- `rpc_complete_task`
- `rpc_return_key_and_create_lock_task`

実装方針:

- RPC が使える場合は RPC を優先します。
- RPC が失敗した場合に備えて、アプリ側フォールバック処理が複数サービスに入っています。

## 13. 現時点の重要な実装メモ

- `item13` / `item14` / `item15` は個別画面に見えても、実質は `SupportDeskScreen` のロール差し替え実装です。
- `item16` は企画者の `eventName` / `eventLocation` を `user_profiles` の `exhibitor_event_name` / `exhibitor_event_location` に保存します。
- `item16` の送信処理では、現状 `eventId` が `null` 固定で組み立てられているため、仕様書上の「開始/終了報告時に `event_id` を必須とする」状態とはまだ差があります。
- 鍵の事前申請では `item16` 側で鍵カタログから候補を作り、必要に応じて `keyMasterService` でマスタ投入を補助します。
- 本部画面には案件対応だけでなく、鍵管理、無線ログ、企画一覧、評価承認など複数タブがあります。
- `npm run web` では地震監視も同時起動するため、管理部統合システムとは別のログが混ざることがあります。

## 14. 変更時の基本方針

- 管理部統合システムの依頼では、まず `item12` から `item16` と `support` を疑う
- その次に `src/services/supabase` の関連サービスを確認する
- DB スキーマ判断が必要なら `docs/AI用プロンプト/supabaseスキーマ参照.md` を読む
- UI は `StyleSheet` だけを使う
- 関数には JSDoc 日本語コメント、変数にもコメントを付ける
- エラーは握りつぶさず、画面導線と保存処理の両方で扱う
