## 1. 最初に押さえること

- このリポジトリは `生駒祭 ERP 2026` の Expo アプリで、大学祭運営向けの業務システムです。
- 現在ここで主に開発しているのは `企画管理部統合システム` です。
- 通常の編集対象は `item12` から `item16`、およびそれに直結する `support` / `src/services/supabase` / 関連ドキュメントです。
- 明示的な依頼がない限り、`item1` から `item10` やその他の独立機能は触らない前提で進めます。
- 現在の実作業では、`docs/管理部統合システム仕様書.md` を最重要仕様として扱います。
- `docs/プロジェクト仕様書.md` にはテンプレート由来の記述も残っているため、管理部統合システムの実装判断では `docs/管理部統合システム仕様書.md` と実コードを優先して整合を見る必要があります。

## 2. 技術スタック

- フロントエンド: React Native + Expo SDK 54
- 実行対象: Web / iOS / Android
- 言語: JavaScript
- バックエンド: Supabase
- 通知: Supabase Edge Functions + Web Push
- 外部連携: Google Apps Script、地震監視サービス
- スタイリング: `StyleSheet` のみ

補足:
- `package.json` では `typescript` 依存が入っていますが、アプリ本体の実装方針は JavaScript です。
- TypeScript は `supabase/functions/` の Edge Functions 側のみ例外的に存在します。

## 4. アプリの全体構造

エントリーフロー:

```text
index.js
  -> App.js
    -> AuthProvider
      -> ThemeProvider
        -> TerminalProvider
          -> FontLoaderProvider
            -> AppNavigator
```

画面遷移の大枠:

```text
AppNavigator
  -> 未認証: LoginScreen
  -> 認証済み: DrawerNavigator
  -> 初回ログイン: PasswordChangeModal / PasswordChangeForm / PasswordSuccessModal
```

Drawer の実装前提:

- `src/navigation/DrawerNavigator.jsx` が全体の主要画面を管理します。
- Web では `768px` を境に、PC は常設サイドバー、モバイルはハンバーガーメニューです。
- 各画面は `ScreenErrorBoundary` でラップされています。

## 8. 実装されている主要フロー

### 8.1 企画者が連絡する

1. `item16` で入力する
2. `exhibitorSupportService` が `support_tickets` を作成する
3. 必要に応じて添付を `ticket_attachments` に保存する
4. `supportWorkflowNotificationService` 経由で通知する
5. 種別によって本部 / 会計 / 物品へ振り分けられる

### 8.2 本部 / 会計 / 物品が対応する

1. `item13` / `item14` / `item15` が `SupportDeskScreen` を使って案件一覧を表示する
2. `supportTicketService` で案件一覧、返信、状態更新を行う
3. 本部は追加で巡回割当、評価承認、鍵管理、無線ログ登録を行う

### 8.3 巡回が対応する

1. `item12` で `patrol_tasks` を取得する
2. `acceptPatrolTask` で「向かいます」を処理する
3. `completePatrolTask` で結果を `patrol_task_results` に記録する
4. 必要に応じて元の連絡案件へメッセージや状態を反映する

### 8.4 鍵返却から施錠確認へつなぐ

1. 本部が `key_loans` の返却処理を行う
2. `returnKeyAndCreateLockTask` が動く
3. `rpc_return_key_and_create_lock_task` が使えれば RPC を優先する
4. 失敗時はアプリ側フォールバックで `patrol_tasks` に施錠確認タスクを作る

## 13. 現時点の重要な実装メモ

- `item13` / `item14` / `item15` は個別画面に見えても、実質は `SupportDeskScreen` のロール差し替え実装です。
- `item14` / `item15` の「対象連絡案件」は `support_tickets` / `ticket_messages` / `ticket_attachments` の Realtime 変更を監視し、手動更新なしでも一覧と詳細を再取得します。
- `item12` から完了件数ランキングは撤去済みで、現在の下部タブは `タスク` / `チェック` / `企画一覧` / `ダッシュボード` の4つに絞っている。
- `item12` の要約表示は `ダッシュボード` タブにだけ集約し、他タブでは上部の大きい概要カードを出さない。巡回担当は必要な画面だけを開いて、`タスク` と `チェック` を中心に操作する前提に寄せている。
- `item12` の定常巡回チェックは `organizations_events` を対象一覧として使い、`organization_name` 順に並ぶ企画候補から1件を選んで記録する。
- `item12` の定常巡回チェックは対象企画を1件選ぶとその企画だけ残して他候補を畳み、`キャンセル` で選び直せる。チェック項目は 5 段階評価ではなく設問ごとの選択肢形式へ変更した。
- `patrol_checks.check_items` は旧来の文字列配列に加え、現行では `{ key, label, answerKey, answerLabel, memo }` の配列も保持する。旧 `score` 形式の履歴も表示側で後方互換対応する。
- `item16` は企画者の `eventName` / `eventLocation` を `user_profiles` の `exhibitor_event_name` / `exhibitor_event_location` に保存します。
- `item16` の履歴表示は双方向チャットではなく、質問内容と管理部から返ってきた回答だけを確認する QA 形式で扱う。
- `item16` の最新連絡案件はタブごとに対象種別を絞って表示し、一覧全体を折りたためる。質問タブは `企画ルール変更` / `配置図変更` / `商品配布基準変更` / `物品破損報告` を表示する。
- `item16` の質問タブでは、最新連絡案件の一覧から案件を選ぶと自動で `質問詳細` までスクロールする。
- `item16` の質問タブでは写真添付を出さず、写真が必要な相談は Discord 連絡へ案内する。緊急呼び出しと開始/終了報告でも添付は扱わず、開始/終了報告フォームは報告種別の選択だけに簡略化した。
- `item16` の質問種別ボタンは、画面上に対応先を表示しない。狭いスマホ幅では 1 列に切り替え、長いラベルでも崩れないようにしている。
- `item16` の送信処理では、現状 `eventId` が `null` 固定で組み立てられているため、仕様書上の「開始/終了報告時に `event_id` を必須とする」状態とはまだ差があります。
- 鍵の事前申請では `item16` 側で鍵カタログから候補を作り、必要に応じて `keyMasterService` でマスタ投入を補助します。
- 本部画面には案件対応だけでなく、鍵管理、無線ログ、企画一覧、評価承認など複数タブがあります。
- `item12` と `item13` の企画一覧タブは `organizations_events` を参照し、団体名の検索付きプルダウンで絞り込むUIにしている。
- `item13` の概況タブでは `企画報告確認` と `施錠確認` をそれぞれ折りたたみでき、`施錠確認` は `担当あり` / `担当なし` / `確認済み` / `未確認` で絞り込める。
- `npm run web` では地震監視も同時起動するため、管理部統合システムとは別のログが混ざることがあります。
- Web Push の OS 通知ポップアップは `public/service-worker.js` が担当し、PWA を閉じている状態でも通知を表示する前提です。
- 管理部統合システムの通知不達は、通知レコード未作成ではなく「受信アカウントの `push_subscriptions` 不足」が主因だった。対策として `item12` / `item13` / `item14` / `item15` / `item16` に非 shared の Push 再登録導線を追加し、画面表示時に購読状態を自己修復する実装を入れている。
- `shared` を編集せずに、返信投稿・ステータス更新・巡回割当の通知は `src/services/supabase/supportNotificationService.js` で補完している。
- `dispatch-notification` は従来の管理者向け個人宛て通知に加えて、連絡案件の返信/状態更新と巡回割当に限り、業務イベント単位で `targetType=user` を許可する。
- 2026-03-07 の追加調査では、配信失敗の直接原因として `dispatch-notification` の Web Push `topic` ヘッダーに `notification.id`（UUID, 36文字）をそのまま入れていた問題を確認した。Web Push の Topic ヘッダー制約に抵触するため、この値は送信オプションから外す。
- Supabase MCP 経由の Edge Function デプロイでは sibling ディレクトリ import がバンドル失敗することがあるため、`dispatch-notification` と `push-subscription` は `_shared` 参照に依存しない単体完結構成を優先する。
- `item15` は画面上の状態を `未対応` / `対応中` / `対応済み` の3段階に集約し、緊急/通常フィルターは出さない。内部ステータスは `new` / `acknowledged` / `in_progress` / `waiting_external` / `resolved` / `closed` のまま保持しつつ UI だけ丸めている。
- `item15` の詳細ではステータス変更を依頼本文より上に出し、Realtime 監視に加えて画面復帰・フォーカス時にも再取得して、通知後に手動更新へ依存しない形にしている。
- `item14` は画面上の状態を `未対応` / `対応中` / `完了` の3段階に固定し、`すべて` / `緊急のみ` / `通常のみ` の絞り込みは出さない。内部保存値は既存 `support_tickets.ticket_status` を使い続ける。
- `item14` の会計向け景品配布基準カードは `prize_distribution.distribution_criteria` を一覧から選び、ポップアップ編集モーダル内で更新できる。一覧の下に常設の編集欄は置かない。
- 連絡案件通知の文面は `src/shared/services/supportWorkflowNotificationService.js` と `src/services/supabase/supportNotificationService.js` で揃え、団体名・依頼者名・企画名・場所が先に分かる順で表示する。
- 2026-03-10 時点で、管理部統合システムの新規案件通知と返信通知はクライアント側でロール ID を解決せず、`dispatch-notification` が `roleNames` を受けてサーバー側で受信者を確定する。会計/物品向け通知はこの経路を通す。
- `item14` / `item15` の `SupportDeskScreen` は `対象連絡案件` / `案件詳細` / `景品配布基準` を折りたためる。案件詳細のステータス変更は確認ポップアップなしで即時更新し、回答送信時の自動 `in_progress` 更新でも通知送信者 ID を渡す。
- `item14` の回答送信では「回答を送信しますか？」確認を出さない。
- `item16` のネイティブ画像添付は `ticketAttachmentService` 側で `ArrayBuffer` 化して Storage へ送る。Expo Image Picker で取得した Blob には直接プロパティを書き込まず、アップロード用メタ情報オブジェクトへ詰め替える。
- `item16` の最新連絡案件は、質問詳細の先頭を含むタイトルで保存し、履歴一覧にも詳細プレビューを2行まで出す。`商品配布基準変更` / `物品破損報告` / `企画ルール変更` が同名で並ばないようにしている。

## 14. 変更時の基本方針

- 管理部統合システムの依頼では、まず `item12` から `item16` と `support` を疑う
- その次に `src/services/supabase` の関連サービスを確認する
- DB スキーマ判断が必要なら `docs/AI用プロンプト/supabaseスキーマ参照.md` を読む
- UI は `StyleSheet` だけを使う
- 関数には JSDoc 日本語コメント、変数にもコメントを付ける
- エラーは握りつぶさず、画面導線と保存処理の両方で扱う
