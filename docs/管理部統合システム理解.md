## 1. このシステムの中核

管理部統合システムは、次の2本柱で運用する設計。

- 連絡案件（`support_tickets`）
  - 出展からの問い合わせ、緊急連絡、鍵申請、開始/終了報告などを一元管理
- 巡回タスク（`patrol_tasks`）
  - 連絡案件や鍵返却を起点に、巡回の受諾・完了・結果入力を管理

## 2. ロールと画面の対応（現実装）

| ロール | 画面 | 主機能 |
| --- | --- | --- |
| 出展（Exhibitor） | `item16` | 連絡登録、履歴確認、追記、添付 |
| 巡回（Patrol） | `item12` | タスク受諾/完了、巡回チェック、未巡回アラート、評価入力 |
| 本部（HQ） | `item13` | 全案件対応、ダッシュボード、巡回割当、鍵管理、評価承認、無線 |
| 会計（Accounting） | `item14` | 会計対象案件の回答/状態更新、景品配布基準編集 |
| 物品（Property） | `item15` | 物品対象案件の回答/状態更新 |

補足:
- `item13`/`item14`/`item15` は共通コンポーネント `SupportDeskScreen` をロール別設定で利用。
- 通常の実装依頼では、この表にある `item12`〜`item16` と `support` を優先して確認する。

## 3. 主要業務フロー（実装ベース）

### 3.1 出展から連絡を送る

1. `item16` で質問/緊急/鍵申請/開始終了のいずれかを送信  
2. `exhibitorSupportService` が `support_tickets` を作成  
3. 種別に応じて通知
   - 会計系: `notify_target=accounting`
   - 物品系: `notify_target=property`
   - それ以外: 本部中心
4. 開始/終了報告は `rpc_create_ticket_and_auto_tasks` で巡回タスク自動生成（RPC未適用時はフォールバック）

### 3.2 巡回が対応する

1. `item12` で未割当または自分担当タスクを取得  
2. 「向かいます」: `acceptPatrolTask`（`rpc_accept_task` 優先）  
3. 「完了」: `completePatrolTask`（`rpc_complete_task` 優先）  
4. 結果は `patrol_task_results` へ記録し、必要時に元連絡案件へメッセージ反映

### 3.3 本部が統制する

1. `item13` で全連絡案件を横断確認  
2. 状態更新、回答、巡回割当、評価承認、無線ログ登録  
3. 鍵返却時は `rpc_return_key_and_create_lock_task` で施錠確認タスク生成

## 6. 2026-02-20 時点の実装メモ

- `item12`〜`item16` の主要導線は実装済みで、仕様書の核フローに概ね対応。
- 通知は「案件作成」「巡回受諾/完了」「施錠確認タスク作成」に接続済み。
- 2026-03-07 時点で、通知不達の一次原因は `dispatch-notification` 側ではなく、管理部統合システム利用アカウントの `push_subscriptions` 不足だった。
- 対策として `shared` を触らず、`item12` / `item13` / `item14` / `item15` / `item16` に Push 再登録カードを出し、画面表示時に Web Push 購読を再同期する導線を追加した。
- 返信投稿、ステータス更新、巡回担当割当の通知は `src/services/supabase/supportNotificationService.js` で補完し、既存 shared 通知ブリッジの外側で不足トリガーを埋めている。
- `dispatch-notification` は管理者向け個人通知に加え、連絡案件の返信/状態更新と巡回割当についてのみ、業務イベントを検証した上で個人宛て送信を許可する形へ拡張した。
- 追加調査で、Web Push 送信オプションに `notification.id`（UUID, 36文字）を `topic` ヘッダーとして渡していたことが全件失敗の直接原因候補と判明した。Topic 制約に抵触するため、この値は使わない。
- `item14` / `item15` の「対象連絡案件」は `support_tickets` / `ticket_messages` / `ticket_attachments` を Realtime 監視し、変更が入ると手動更新なしで再取得される。
- `item12` の巡回サポートと `item13` の本部サポートにある「企画一覧」は `organizations_events` を参照し、団体名の検索付きプルダウンで対象団体を絞ってから企画を確認する。
- `item13` の概況タブでは `企画報告確認` と `施錠確認` を個別に折りたためる。`施錠確認` は `担当あり` / `担当なし` と `確認済み` / `未確認` の組み合わせで絞り込める。
- `item12` では完了件数ランキングを廃止済みで、現在の下部タブは `タスク` / `チェック` / `企画一覧` / `ダッシュボード` の4つに整理している。
- `item12` の要約情報は `ダッシュボード` タブだけに置き、他タブでは概要カードを繰り返さない。巡回担当は要約を一度見たら、そのまま `タスク` または `チェック` に移って操作する構成にしている。
- `item12` の定常巡回チェックは `organizations_events` を巡回対象マスタとして参照し、`organization_name` 順に並んだ企画から1件を選んで登録する。
- `item12` の定常巡回チェックは対象企画を選ぶとその企画だけ残して他候補を畳み、`キャンセル` で選び直す。チェック項目は 5 段階評価から設問別の選択肢入力へ切り替えた。
- `patrol_checks.check_items` は現在、各項目について `key / label / answerKey / answerLabel / memo` を持つ JSON 配列でも保存する。旧 `score` 形式や文字列配列は履歴表示だけ後方互換で読む。
- Web Push のポップアップ表示は `public/service-worker.js` が担当し、PWA を閉じている状態でも OS 通知として表示する前提にしている。
- `item16` の履歴画面は双方向チャットではなく、質問内容と管理部からの回答だけを見る QA 形式へ寄せている。
- `item16` の最新連絡案件はタブ別に `質問` / `緊急` / `鍵申請` / `開始終了` の対象種別だけを表示し、一覧全体を折りたためる。質問タブでは `企画ルール変更` / `配置図変更` / `商品配布基準変更` / `物品破損報告` の4種を出す。
- `item16` の質問タブでは、最新連絡案件の一覧から案件を選ぶと `質問詳細` まで自動スクロールする。
- `item16` の質問タブでは写真添付を出さず、写真が必要な相談は Discord 連絡へ案内する。緊急呼び出しと開始/終了報告でも添付は扱わず、開始/終了報告は報告種別の選択だけ送る形に簡略化した。
- `item16` の質問種別ボタンは対応先を表示せず、狭いスマホ幅では 1 列表示へ切り替えて長いラベルでも崩れないようにしている。
- `item14` の見た目上の状態は `未対応` / `対応中` / `完了` の3段階に固定し、`すべて` / `緊急のみ` / `通常のみ` の絞り込みは表示しない。内部保存値は既存 `support_tickets.ticket_status` を使い続ける。
- `item14` では会計担当が `prize_distribution` 一覧から対象の景品配布基準を選択し、ポップアップ編集モーダルで `distribution_criteria` を更新する。対象が多い日でも一覧を崩さず、その場で保存できる。
- `item15` の見た目上の状態は `未対応` / `対応中` / `対応済み` の3段階に固定し、緊急/通常フィルターは表示しない。内部保存値は既存 `support_tickets.ticket_status` を使い続ける。
- `item15` の詳細画面ではステータス変更を依頼本文より上に置き、通知到着後も画面復帰・フォーカス・Realtime で自動再取得する。
- 連絡案件通知は新規作成時も返信/状態更新時も、団体名・依頼者名・企画名・場所が先に見える文面へ統一した。
- 送信先ロールは `notify_target=property` が `物品部 + 管理者`、`notify_target=accounting` が `会計部 + 管理者`、それ以外が `企画管理部 + 管理者`。`emergency` / `start_report` / `end_report` は `警備部 + 企画管理部 + 管理者` に広げる。
- 2026-03-10 の修正で、新規案件通知と返信通知は `dispatch-notification` の `roleNames` 解決へ統一した。会計/物品宛ての新規通知は `supportWorkflowNotificationService`、返信/状態更新は `supportNotificationService` から同じ受信者決定ロジックを使う。
- `item14` / `item15` の案件詳細は、ステータス変更を確認ポップアップなしで即時反映する。回答送信時に自動で `in_progress` へ上げる場合も `notifyActorUserId` を渡し、通知文面と送信者判定を崩さない。
- `item14` / `item15` は `対象連絡案件` と `案件詳細`、`item14` は加えて `景品配布基準` を折りたためる。作業量が多い日でも一覧と編集エリアを切り替えながら扱う前提にしている。
- `item14` の回答送信は確認ダイアログなしで即送信する。
- `item16` の画像添付は React Native の Blob をそのまま Storage に渡さず `ArrayBuffer` 化して送る。最新連絡案件にはタイトルと一覧プレビューの両方に質問詳細の先頭を載せ、似た種別が並んでも判別しやすくした。
- 仕様差分として要確認:
  - `item16` 送信時の `eventId` が現状 `null` 固定（仕様書では開始/終了報告時の `event_id` 必須）。
  - 設置型/持ち歩き分離は、現状「同一アプリ内のロール別画面」で実現。

## 8. 2026-02-23 通知403の原因と対処

- 現象: `item16` から連絡案件を送信した際に `dispatch-notification` が `403 Forbidden` を返し、通知送信のみ失敗していた。
- 原因: Edge Function `dispatch-notification` 側の認可条件が実運用フローより厳しく、企画制作部アカウントの通知送信が拒否されていた。
- 対処: 認証済みユーザーの `targetType=roles` 通知を許可し、`targetType=user` は管理者ロールのみ許可する形へ修正。
- 反映: Supabase Edge Function `dispatch-notification` を version 8 としてデプロイ済み。
- 補足: 通知本文・受信者作成・Push送信の既存フローは維持。

- 目的: 企画者（item16利用者）アカウントの追加作成。
- 実施結果:
  - Supabase MCP 接続は OAuth 更新エラーで失敗し、MCP経由のDB操作は未実行。
  - Supabase Auth `signUp` は成功（`test-exhibitor-20260227@gmail.com` 作成）。
  - ただしメール確認必須設定のため、確認前は `signInWithPassword` が `Email not confirmed` で失敗。
- 制約:
  - `user_roles` は仕様上 `SELECT` のみ公開で、匿名キー/一般ユーザー経由のロール付与は不可。
  - 企画者として即時利用するには、管理権限（service role またはダッシュボード管理者）で
    1) メール確認完了、2) `user_roles` へ企画者ロール付与、が必要。

## 2026-03-09 追記

- 権限方針を更新した。`item12` は `企画管理部 / 警備部 / 管理者`、`item14` は `会計部 / 管理者`、`item15` は `物品部 / 管理者`、`item13` は `管理者` のみを想定する。
- 企画管理部は巡回サポート中心の運用に寄せ、本部サポートは管理者確認用として扱う。
- `item14` / `item15` の通知はそれぞれ会計部・物品部を主対象にし、巡回系通知は `企画管理部 / 警備部 / 管理者` に送る。
- `item16` は QA 表示を維持したまま FAQ / 入力ヒントを外し、最新の連絡案件だけ利用者側でクローズできるようにした。
- 写真添付は `expo-image-picker` を追加し、ネイティブ端末からも写真ライブラリ経由で送信できるようにした。
- `dispatch-notification` は 403 調査をしやすくするため、会計/物品/本部/巡回/管理者のロール別名を認可対象へ広げ、個人宛て通知が拒否されたときは送信者・宛先・案件ID・ロール一覧を Edge Function ログへ残すよう更新した。会計/物品の回答通知とステータス更新通知を直すには、この関数の再デプロイが前提になる。
- 通知権限の仕様はさらに緩め、現在は `dispatch-notification` に有効な Bearer トークンまたは internal token が渡っていれば、管理部統合システムの業務種別に関係なく `user` / `roles` の両方へ送信できる前提に変更した。

## 2026-03-10 追記

- `item14` / `item15` の部署向け `SupportDeskScreen` では、案件一覧で案件を選択すると `案件詳細` まで自動スクロールするようにした。
- `item14` / `item15` の案件詳細ステータス更新は Picker ではなく 3 状態のボタン選択で更新する。
- 会計対応画面・物品対応画面の冒頭説明文は撤去し、物品対応画面の添付補足文も出さない。
- 本部サポートの `巡回タスク割当` は `企画管理部` ユーザーのみを担当者候補に出す。
