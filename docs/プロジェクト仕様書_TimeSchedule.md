# プロジェクト仕様書 TimeSchedule

最終更新日: 2026/03/27
ステータス: 仕様確定（実装準備）

---

## 1. 概要

### 1.1 目的
大学祭当日の時刻進行を可視化し、5分単位で「どの企画が開催中か」を確認できる画面を提供する。

### 1.2 対象範囲
- 表示対象は全企画ではなく、指定エリアで開催される企画のみ
- 日ごとの時間表を表示
- 5分ごとのタイムスロットで開催企画を表示

### 1.3 対象ユーザー
- 企画管理部（本部）
- 必要に応じて巡回担当

### 1.4 表示対象エリア
- 実学ホール
- 11月ホール
- 記念会館
- 人工芝グラウンド
- その他

## 2. 機能要件

### 2.1 日付切替
- 日付単位で時間表を切替できる
- 初期表示は祭1日目
- 前回表示日を端末ローカルに記憶し、次回起動時は前回表示日を復元する
- 前回表示日が保存されていない初回のみ、祭1日目を表示する

### 2.2 時間表表示
- 運用時間は 9:00 〜 20:00
- タイムスロットは5分刻み（9:00, 9:05, ..., 19:55）
- 各スロットに該当する開催中企画を表示する
- 同時刻に複数企画がある場合は横スクロールカードで表示する

### 2.3 表示対象の判定条件
- カテゴリではなく開催場所（エリア）で表示対象を判定する
- 表示対象エリア外の企画は時間表に出さない

### 2.4 企画表示情報
- 企画名
- 開始時刻
- 終了時刻
- 開催場所（必須表示）

### 2.5 並び順
- 同一スロット内は開始時刻昇順
- 開始時刻が同じ場合は企画名昇順

### 2.6 絞り込み機能
- エリア絞り込みを提供する
- 複数エリアの同時選択に対応する
- 絞り込み条件は端末ローカルに保存し、アプリ再起動後も復元する

### 2.7 更新方式
- 画面ロード時のみデータ取得を実行する
- 時間ごとの自動更新は行わない
- 必要に応じて手動更新（再取得ボタン）を提供する

### 2.8 企画カード詳細表示
- 企画カードタップ時は詳細モーダルを表示する
- 詳細モーダルは企画屋台一覧の既存詳細モーダルと完全同一仕様を使用する
- 表示項目・レイアウト・操作（閉じる操作を含む）は企画屋台一覧の既存仕様に一致させる

---

## 3. 画面要件

### 3.1 画面名
- TimeSchedule

### 3.2 基本構成
- ヘッダー（画面名）
- 日付切替エリア
- 時間表本体（5分単位）
- 絞り込み表示（エリア）

### 3.3 レイアウト
- 時間軸は縦方向
- 各時刻行の企画表示は横スクロールカード
- 企画カードには「企画名 / 開始-終了 / エリア」を表示

### 3.4 レスポンシブ
- モバイル・タブレット・Webで崩れないこと
- モバイルでは1行あたりカードを見切れ表示にしてスワイプ誘導する
- Webではカード幅を固定し、空白時は「該当なし」を表示する

## 4. データ要件

### 4.1 想定データソース
- 現状 `events` と `stalls` に開催日カラムが存在しないため、DB構造の見直しが必要
- TimeSchedule は日付軸での表示が前提のため、開催日を持つスケジュールデータを別管理する

### 4.2 必須データ項目
- 企画ID
- 企画種別（event / stall）
- 企画名
- 開催日
- 開始時刻
- 終了時刻
- 開催場所コード（area_code）

### 4.3 データ整合
- 開始時刻 < 終了時刻 を満たすこと
- 開催場所は `area_code` で管理し、許可コード以外は登録不可とする
- 日跨ぎイベントは表示対象日の範囲に切り詰めて表示する（当日分のみ）

### 4.4 DB再設計方針（確定方針）
- 新規テーブル `event_schedule_slots` を追加する
- `events` / `stalls` 本体は既存通り維持し、日程は `event_schedule_slots` に分離する
- `events` の旧時間カラム（`start_time` / `end_time` / `start_times` / `end_times`）は廃止し、TimeScheduleの時間参照は `event_schedule_slots` に一本化する
- 複数日開催かつ日ごとに異なる時間帯は `event_schedule_slots` を日付ごとに複数行持つことで表現する
- `event_schedule_slots` の主カラム案
	- `id` (uuid, PK)
	- `source_type` (text: `event` or `stall`)
	- `source_id` (text: 元テーブルIDを文字列で保持)
	- `schedule_date` (date)
	- `start_time` (time)
	- `end_time` (time)
	- `area_code` (text)
	- `is_visible_time_schedule` (boolean, default true)
	- `created_at` / `updated_at` (timestamptz)
- TimeSchedule 画面は `schedule_date` と `area_code` で抽出し、5分スロットへ投影する

#### source_id を text にする理由
- `events` と `stalls` の ID 型差異に依存せず統一的に扱える
- `source_type + source_id` の組み合わせで参照先を一意に識別できる
- アプリ側の集約クエリ実装を単純化できる
- 注意点: 厳密な外部キー制約は貼れないため、登録・更新時に存在チェックをサービス層で実施する

#### events 連携ルール
- `source_type='event'` のレコードは `events.id::text = source_id` で参照する
- DBトリガーで `events` / `stalls` の参照存在チェックを行い、不正な `source_id` の登録を防止する
- `events` に参照用キャッシュカラム `schedule_dates` / `schedule_start_times` / `schedule_end_times` を持たせる
- 一覧画面と詳細モーダルは `events.schedule_start_times` / `events.schedule_end_times` を参照して表示する

### 4.5 複数日・日別時間差の管理ロジック
- 1つの企画に対して、開催日ごとに1レコード以上を登録する
- 同じ企画でも日によって開始/終了時刻が異なる場合は、日付ごとに別レコードで管理する
- 同一日の中で複数回開催する場合は、同一 `source_type + source_id + schedule_date` で複数レコードを許容する
- 取得クエリは以下条件で当日分のみ抽出する
	- `schedule_date = 選択日`
	- `is_visible_time_schedule = true`
	- `area_code in (選択中エリア)`
- スロット投影判定は `start_time <= slot_time < end_time` を採用する
- 日跨ぎデータ（例: 19:30-21:00）は運用時間と当日範囲で切り詰める
	- 表示開始: `max(start_time, 09:00)`
	- 表示終了: `min(end_time, 20:00)`

### 4.6 エリアマスタ管理（確定仕様）

- エリアは自由入力を禁止し、`area_code` で管理する
- 表示名はマスタテーブルから解決する
- コード一覧:
	- `JITSUGAKU_HALL`（実学ホール）
	- `NOVEMBER_HALL`（11月ホール）
	- `KINEN_KAIKAN`（記念会館）
	- `ARTIFICIAL_TURF`（人工芝グラウンド）
	- `OTHER`（その他）

#### 4.6.1 追加テーブル案
- テーブル名: `time_schedule_areas`
- 主カラム案:
	- `area_code` (text, PK)
	- `area_name` (text, not null)
	- `display_order` (integer, not null)
	- `is_active` (boolean, default true)
	- `created_at` / `updated_at` (timestamptz)

#### 4.6.2 リレーション
- `event_schedule_slots.area_code` -> `time_schedule_areas.area_code`
- 画面の絞り込みUIは `time_schedule_areas` を参照して表示順を決定する
