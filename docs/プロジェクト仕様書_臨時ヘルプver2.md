## 4. 画面仕様

## 4.1 画面構成

### 管理者タブ

- 募集作成
- 募集一覧
- 募集履歴

### 一般ユーザータブ

- 募集一覧
- 応募済み

## 4.2 募集一覧（一般ユーザー）

ヘッダー操作（左から）:

1. 部署フィルタ（プルダウン）
2. ソート（プルダウン）
3. 再読み込み

部署フィルタ仕様:

- 値: `head_organization`（作成者所属組織）を元に動的生成
- 候補: `すべての部署` + 現在表示中募集に存在する組織のみ
- 選択中の部署が候補から消えた場合は `すべての部署` に自動復帰

ソート仕様:

- `新しい順`（`created_at` 降順）
- `古い順`（`created_at` 昇順）
- `募集人数の多い順`（`headcount` 降順）
- `募集人数の少ない順`（`headcount` 昇順）

## 4.3 募集一覧（管理者）

- `open` 募集 + `auto_full`/`auto_date_passed` の自動クローズ募集を表示
- カードから編集 / 終了 / 再開 / 応募者一覧開閉 / 削除（作成者のみ）を操作可能
- `auto_full` は「募集人数到達済み」バッジ表示
- `auto_date_passed` は「募集日経過」バッジ表示
- `auto_full` の `closed` 募集では「再開」ではなく「募集を終了」ボタンを表示し、`close_reason=manual` へ確定させる

## 4.4 募集履歴（管理者）

- `includeClosed=true` で取得した履歴一覧を表示（論理削除済みは除外）
- 募集一覧と同様に編集・終了・再開・削除等の操作が可能

## 4.5 応募済み（一般ユーザー）

- 自分が応募した募集を表示
- 応募取り消しボタンを表示

## 5. 募集データ仕様

## 5.1 主なカラム（`rinji_help_recruits`）

- `id`
- `department_id`
- `head_user_id`
- `headcount`
- `work_date`
- `work_time`
- `location`
- `meet_time`
- `meet_place`
- `description`
- `reward`
- `belongings`
- `status` (`open` / `closed`)
- `close_reason` (`manual` / `auto_full` / `auto_date_passed`)
- `created_at`
- `updated_at`

## 5.2 表示拡張項目（アプリ側で付与）

- `applicant_count`（応募数）
- `head_organization`（作成者所属組織）

## 5.3 `description` の内部フォーマット

`description` は1つの文字列に以下を連結して保持する。

1. 募集タイトル
2. 業務内容
3. 途中参加可否メタ (`allow` / `deny`)

区切り:

- タイトル区切り: `\n\n---\n\n`
- メタ区切り: `\n\n::META::\n\n`

例:

```text
募集タイトル

---

業務内容本文

::META::

allow
```

## 5.4 論理削除マーカー

- 削除時は `description` に `\n\n::DELETED::\n\n1` を付与
- 一覧取得時にマーカー付き募集は除外する

## 6. 募集作成・編集仕様

## 6.1 入力項目

- 募集人数（必須）
- 場所（必須）
- 集合場所（任意）
- 募集日（必須、現実装は固定選択肢）
- 開始時刻（必須）
- 所要時間目安（任意）
- 報酬（必須）
- 集合時間（任意）
- 持ち物（任意）
- 募集タイトル（必須）
- 業務内容（必須）
- 途中参加可否（可 / 不可）

## 6.2 バリデーション

必須チェック:

- `headcount`
- `location`
- `work_date`
- `work_time`（開始時刻）
- `title`
- `description`
- `reward`

## 6.3 自動補完

- `meet_place` 未入力時: `location` を採用
- `meet_time` 未入力時: 開始時刻を採用
- `belongings` 未入力時: `なし`

## 6.4 送信ペイロード制御

作成時:

- `notify_all_on_create`（トグル）
- 初期値: `false`

編集時:

- `notify_applicants_on_update`（トグル）
- 初期値: `true`

## 6.5 時刻入力と保存ルール

- `work_time` は `開始〜終了` 形式で保存する
- 所要時間目安が入力されている場合:
  - `開始時刻 + 所要時間` で終了時刻を算出して保存
- 所要時間目安が未入力の場合:
  - `開始時刻〜終了時刻未定` で保存
- 「現在時刻」クイック選択:
  - 押下時点の `HH:mm` を開始/集合時刻へセットして保存
- `meet_time` 未入力時:
  - 集合時間は開始時刻を利用する

## 7. 募集状態遷移

## 7.1 ステータス

- `open`: 募集中
- `closed`: 受付終了

## 7.2 `close_reason`

- `manual`: 手動終了または削除処理
- `auto_full`: 募集人数到達で自動終了
- `auto_date_passed`: 募集日経過で自動終了

## 7.3 遷移ルール

- 応募時に定員到達 -> `closed/auto_full`
- `close_expired_rinji_recruits()` 実行時に募集日経過 -> `closed/auto_date_passed`
- 管理者終了操作 -> `closed/manual`
- 再開操作 -> `open`（`close_reason` は `null`）
  - ただし応募人数 >= 募集人数の場合は再開不可

## 7.4 応募取り消し時の自動再開

- 対象: `closed/auto_full` のみ
- 応募取り消しで応募数が定員未満になった場合に `open` へ自動再開

## 8. 応募仕様

## 8.1 応募登録

- `rinji_help_applications` に `recruit_id` + `applicant_user_id` を追加
- 重複応募は DB 一意制約エラーで防止
- 通常応募時に重複エラーが返った場合は失敗扱いとし、`すでに応募済みです。応募済みタブをご確認ください。` を表示

## 8.2 応募取り消し

- 該当 `recruit_id` + `applicant_user_id` を削除
- 取り消し成功時のみ通知処理を実行

## 8.3 応募者一覧

- 管理者のみ表示可能
- 表示項目:
  - 応募者所属
  - 応募者名
  - 応募日時（`YYYY/MM/DD HH:mm`）

---

## 9. 削除仕様（論理削除）

- 物理削除は行わない
- 募集作成者のみ削除可能
- `rinji_help_applications` の応募レコードは物理削除しない（募集側を論理削除して一覧から除外）
- 実行時:
  - `status=closed`
  - `close_reason=manual`
  - `description` に削除マーカー付与

確認ダイアログ:

- 応募者なし:  
  `この募集を削除します。削除後は一覧に表示されなくなります。よろしいですか？`
- 応募者あり:  
  `この募集を削除します。削除後は一覧に表示されなくなり、応募者情報も削除されます。よろしいですか？`

## 10. 通知仕様（現行）

通知送信は `sendNotificationToUser`（`dispatch-notification` Edge Function経由）を利用する。

## 10.1 通知イベント一覧

1. 募集作成通知（任意）
- トリガ: 募集作成 + `notify_all_on_create=true`
- 宛先: 作成者以外の全ユーザー
- タイトル: `新しい臨時ヘルプ募集が作成されました`
- 本文: 募集タイトル / 募集日時 / 場所

2. 応募発生通知
- トリガ: 応募登録
- 宛先: 募集作成者
- タイトル: `[臨時ヘルプ]「募集タイトル」に応募がありました`
- 本文: 募集タイトル / 応募者所属 + 応募者名

3. 応募取り消し通知
- トリガ: 応募取り消し
- 宛先: 募集作成者
- タイトル: `[臨時ヘルプ]「募集タイトル」の応募が取り消されました`
- 本文: 募集タイトル / 取消者所属 + 取消者名

4. 募集更新通知（任意）
- トリガ: 募集編集 + `notify_applicants_on_update=true`
- 宛先: その募集の応募者（更新者本人は除外）
- タイトル: `[臨時ヘルプ]「募集タイトル」の募集内容が更新されました`
- 本文: 差分要約
  - タイトル/人数/日付/開始時刻/場所/集合場所/集合時間/途中参加可否
  - 業務内容は本文全文ではなく `業務内容欄が変更されました` を表示

5. 定員到達通知
- トリガ: `open -> closed/auto_full` への遷移
- 宛先: 募集作成者（自操作は通知しない）
- タイトル: `[臨時ヘルプ] 募集が定員に達しました`
- 本文: `応募人数が募集人数に到達しました（応募数/募集数）`

6. 募集削除通知
- トリガ: 募集削除
- 宛先: その募集の応募者（削除実行者は除外）
- タイトル: `[臨時ヘルプ]「募集タイトル」の募集が削除されました`
- 本文:  
  `募集が削除されたため、この募集への参加は不要になりました`  
  + 募集名 / 募集日時 / 場所

## 10.2 通知共通ルール

- 自分の操作で自分に通知しない
- 送信失敗は `console.warn` で記録し、主処理は継続

## 11. 通信エラーと再送仕様

対象操作:

- 募集作成
- 応募

仕様:

- 通信系エラー時はリクエストをキューに保存（`AsyncStorage`）
- キューキー:
  - 作成: `create:userId:payload`
  - 応募: `apply:recruitId:userId`
- 最大3回自動再送、間隔2秒
- 再送中/再試行待ちのメッセージを画面上部に表示
  - `通信エラーのため{募集作成|応募}を再送中です（n/3）`
  - `通信エラーのため{募集作成|応募}の再送に失敗しました。2秒後に再試行します（n/3）`
- 成功時は成功トースト表示
- 失敗上限到達時は失敗メッセージ表示

永続化:

- ストレージキー: `item8_rinji_help_retry_queue_v1`
- 画面再表示時に未送信キューを復元して再送を再開
- 復元時は `未送信のリクエストを検出したため、自動で再送を試行します。` を表示

## 11.1 再送時の重複応募扱い（冪等性）

- 応募の再送時は `treatDuplicateAsSuccess=true` で実行
- 理由:
  - 通信断で「DB登録は成功したがクライアントが失敗扱い」になったケースを救済するため
- 挙動:
  - 再送で重複エラーになっても成功として扱い、一覧再取得を行う
  - 応募通知・定員到達通知の後続処理も実行する
