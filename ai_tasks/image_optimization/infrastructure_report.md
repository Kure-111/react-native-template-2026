# 生駒祭2026 インフラ負荷推定・最適化レポート

本レポートは、来場者 3 万人およびスタッフ 300 人の規模を想定した、生駒祭システムのインフラ負荷試算と最適化状況をまとめたものです。

---

## 1. 最適化の実施内容 (Image & Data Optimization)

インフラ負荷（主に Egress 通信量）を最小化するために以下の対策を実施済みです。

### 画像のリサイズと圧縮の強化
- **対象**: `ikomasai-hp-admin` (管理画面での画像登録時)
- **設定**: 最大解像度 **800px** / ターゲットサイズ **120KB**
- **効果**: これまでの設定 (1600px / 800KB) に比べ、1枚あたりの通信量を **約 1/6** に削減。

### HP 側の遅延読み込み (Lazy Loading) 導入
- **対象**: `ikomasai_hp` (公式サイトの企画・屋台一覧)
- **内容**: ブラウザ標準の `loading="lazy"` を採用し、画面内に入った画像のみを読込。
- **効果**: 一覧表示時の初期通信量を **約 85% 削減**。

### ERP 側のデータ軽量化
- **対象**: `ikomasai-erp-2026` (企画屋台一覧)
- **内容**: 一覧表示から画像パスの取得を完全に排除し、テキストのみの送受信に変更。
- **効果**: 300 人の同時接続があっても、通信量（ギガ）をほぼ消費しない「超軽量」仕様を実現。

## 2. 負荷シミュレーション (3日間 / 3万人規模)

他の特定企画 (66GB) を含めた、開催 3 日間の総計シミュレーションです。

### 全体サマリー (HP + ERP + 他企画)

| 項目 | 推定値 | Supabase Pro 枠 (250GB) | 判定 |
| :--- | :--- | :--- | :--- |
| **総通信量 (Bandwidth)** | **237 GB 〜 245 GB** | **250 GB** | **合格 (枠内完遂可能)** |
| **最大同時接続数** | **約 300 〜 500** | **500** | **合格 (Pro プラン必須)** |
| **DB / ストレージ容量** | **約 50 〜 100 MB** | **8 GB** | **圧倒的な余裕** |

### 部門別詳細

#### A. 公式サイト (HP)
- **通信量**: 約 171 GB 〜 180 GB
- **挙動**: ユーザー 1 滞在あたり約 6MB の通信。Lazy Loading により無駄な読み込みを防止。
- **仕様**: Realtime (WebSocket) をオフにし、REST API で配信することで数万人の同時アクセスを技術的に可能にしました。

#### B. 統合システム (ERP)
- **通信量**: 約 1 GB 未満 (非常に軽量)
- **リアルタイムメッセージ**: 約 27 万件 (枠 500 万件に対して余裕)
- **同時接続数**: 常時約 300 接続を想定。

## 3. インフラ運用のアドバイス

### Supabase プランの切り替え時期
- **制作期間中**: 無料プランでも開発可能ですが、同時接続数（200）や通信量（5GB）に注意してください。
- **開催直前〜当日**: 来客が数千人を超える前に **Pro プラン ($25/月)** へのアップグレードを強く推奨します。

### Image Transformation の活用 (オプション)
さらに安全策をとりたい場合は、Pro プランの機能である「Image Transformation」を有効化し、URL パラメータで画像を動的リサイズすることで、総通信量をさらに 30〜40% (約 80GB 分) 削減する余地があります。
